# 规范工程模板

> **来源**：四层技能框架调研
> **用途**：任何超过1小时的任务、需要AI自主运行的指令，必须用此格式
> **五原语**：问题陈述 / 验收标准 / 约束 / 分解 / 评估

---

## 模板

```markdown
# [SPEC] {任务名称}

## 问题陈述
{一句话描述：要解决什么问题？为什么重要？}

## 验收标准
- [ ] {可观察、可衡量的成功条件1}
- [ ] {可观察、可衡量的成功条件2}
- [ ] {可观察、可衡量的成功条件3}

## 约束
- **MUST**：{必须做的事}
- **MUST NOT**：{绝对不能做的事}
- **PREFER**：{有多个选项时优先选择}

## 分解
1. {子任务1——预估时间}
2. {子任务2——预估时间}
3. {子任务3——预估时间}

## 评估
成功 = {定义什么算完成}
失败 = {定义什么算失败，以及失败时的处理}
```

---

## 示例：知识库消费通路验证

```markdown
# [SPEC] 知识库消费通路验证

## 问题陈述
新窗口启动时，能否通过MCP检索到之前校准的知识？如果不能，之前的积累就白费了。

## 验收标准
- [ ] chroma_list_collections返回包含knowledge_base_v2
- [ ] 检索"碰撞校准"返回R1-V16
- [ ] 检索"角色膨胀"返回A12
- [ ] 检索"上下文衰减"返回B45

## 约束
- **MUST**：使用已配置的chroma-mcp
- **MUST NOT**：修改用户级MCP配置
- **MUST NOT**：假设成功，必须先验证再报告
- **PREFER**：验证失败时尝试至少3种不同关键词

## 分解
1. 列出当前collection（5分钟）
2. 执行3组语义检索测试（10分钟）
3. 验证返回结果是否包含目标条目（5分钟）
4. 记录结果，如有问题标记待修复（5分钟）

## 评估
成功 = 4项验收标准全部通过
部分成功 = 2-3项通过，记录失败原因
失败 = ≤1项通过，暂停并报告，不继续执行
```

## 示例：把现有INFRA指令改写为规范格式

### 原始格式（自由文本）
```
INFRA-003：验证chroma-mcp是否正常工作
```

### 改写后（规范格式）
```markdown
# [SPEC] INFRA-003 验证chroma-mcp工作状态

## 问题陈述
chroma-mcp配置后从未验证过实际效果，需要确认MCP工具可用、数据库可达、检索功能正常。

## 验收标准
- [ ] CallMcpTool调用chroma-mcp的list_collections成功
- [ ] 返回结果包含knowledge_base_v2
- [ ] CallMcpTool调用query_collection成功
- [ ] 返回结果与预期条目匹配

## 约束
- **MUST**：使用现有MCP配置，不新建
- **MUST NOT**：修改数据库内容
- **PREFER**：出错时记录错误信息而非静默失败

## 分解
1. 调用list_collections（2分钟）
2. 解析返回结果（2分钟）
3. 调用query_collection测试检索（5分钟）
4. 验证返回条目（3分钟）
5. 输出验证报告（3分钟）

## 评估
成功 = 全部验收标准通过
失败 = 任一调用报错或返回不符合预期
失败处理 = 记录错误，输出诊断建议
```

## 什么时候用

| 场景 | 是否必须用 |
|-----|----------|
| 预估超过1小时的任务 | ✅ 必须 |
| AI需要自主运行多步的任务 | ✅ 必须 |
| 涉及多文件/多工具的操作 | ✅ 必须 |
| 简单单步操作 | ❌ 不需要 |
| 对话式碰撞 | ❌ 不需要 |

---

## 为什么需要

来自四层框架调研：
- 智能体35分钟后性能下降
- 任务持续时间翻倍→失败率翻四倍
- 必须通过结构化规范把长任务分解为可管理的块

来自B45（我们的知识库）：
- 上下文越长越模板化
- 质量衰减是结构性宿命
- 规范工程是对抗衰减的方法
