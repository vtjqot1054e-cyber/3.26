# SK-09 生态建模

> 职责：逐步构建对"用户+公司+行业"整体生态的理解

---

## 核心理念

1. **生态不是问出来的，是在对话中浮现的**
2. **认知是逐步积累的，不是一次性完成的**
3. **生态是动态的，模型需要持续演化**

---

## 元原则：为了解决问题，必须先知道什么

为了帮用户解决任何问题，必须建立**三层认知结构**：

```
第一层：大背景（宏观）
├── 时间/阶段：现在是什么时候？什么周期？
├── 空间/地域：在哪里发生？
├── 主体范围：谁的事？一个人还是几个人？
├── 场景性质：日常还是关键场合？
└── 利益相关方：谁都在这个局里？

第二层：问题本身（中观）
├── 现状：现在是什么情况？
├── 目标：想要什么？"更好"是什么意思？
├── 障碍：为什么现在没达到？
├── 资源：有什么可用的？
└── 约束：不能/不愿做什么？

第三层：外部生态（微观/细节）
├── 相关方分类：客户/对手/合作方各有哪些类型？
├── 每类的需求和特征
├── 连接方式：他们之间怎么互动？
└── 规则和趋势：游戏规则是什么？在变吗？
```

**三层是嵌套关系**：大背景包含问题本身，问题本身连接外部生态。

**处理问题时**：
1. 先检查三层信息是否足够
2. 不够的标注缺失
3. 在后续对话中逐步补充
4. 信息足够后再给建议

## 生态结构

### 一、内部生态（用户+公司）

```
用户个人
├── 核心画像 → /数字分身/核心画像.md
├── 动态状态 → /数字分身/动态状态.md
├── 能力边界：能做什么、不能做什么
├── 配置限制：底层不支持什么
└── 价值观约束：不愿做什么

公司/团队
├── 成员档案 → /人物档案/
├── 组织架构：谁管谁、谁配合谁
├── 核心能力：公司能交付什么
├── 资源盘点：人脉、资金、工具、牌照
└── 约束条件：成本、人力、时间

业务能力
├── 能做的业务类型
├── 做得好的（核心竞争力）
├── 做得一般的
└── 做不了的
```

### 二、外部生态（行业）

```
客户生态
├── 客户类型分类（如：学会/科室/专家/厂商）
├── 每类客户的：
│   ├── 典型结构（谁决策、谁执行、谁受益）
│   ├── 真实需求（不是表面需求）
│   ├── 决策模式（怎么选供应商）
│   └── 痛点和偏好
└── 客户之间的关系网络

竞争对手生态
├── 主要竞争对手是谁
├── 他们的模式（怎么获客、怎么服务）
├── 他们的优势
├── 他们的弱点
└── 他们与客户的连接方式

行业环境
├── 显规则（合规要求、招投标规则）
├── 潜规则（真正的游戏规则）
├── 变化趋势（行业在往哪走）
└── 风险点（什么可能出事）
```

### 三、连接点

```
能力-需求匹配
├── 用户的能力能满足哪类客户的哪个需求
├── 哪些需求用户满足不了
└── 哪些能力目前没有对应的客户

竞争定位
├── 用户与竞争对手的差异点
├── 用户的护城河在哪（如果有）
└── 用户的软肋在哪

关系资产
├── 用户在哪些节点有关系
├── 哪些关系是用户自己的
├── 哪些关系是托管在别人手里的
└── 关系的强度和稳定性
```

## 信息收集机制

### 在对话中捕捉

| 用户说了什么 | 归档到哪里 |
|--------------|------------|
| 提到客户类型/特征 | 客户生态 |
| 提到竞争对手 | 竞争对手生态 |
| 提到行业规则/潜规则 | 行业环境 |
| 提到公司能做/不能做的 | 业务能力 |
| 提到团队成员 | 人物档案 |
| 提到自己的限制 | 核心画像 |

### 主动探测

当讨论业务问题时，检查：
- 这个问题涉及生态的哪个部分？
- 那个部分的信息够不够？
- 如果不够，需要追问什么？
