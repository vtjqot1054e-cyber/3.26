## 二、5 个可借鉴机制的判断质量

### 5.1 遗忘曲线（agent-memory-skill）

报告说"与我们高度契合"——**验证通过，但有一个报告没提到的关键细节**：

agent-memory-skill 的 decay 公式是 `relevance = max(0.1, 1.0 - days × 0.015)`，这是**线性衰减**。真正的 Ebbinghaus 曲线是指数衰减。对我们来说线性够用（简单），但报告把它等同于"遗忘曲线"在学术上不严谨。

**数字真我约束**：改 frontmatter 结构 = 改数据格式 = JSON 镜像和向量库同步脚本都要跟着改。报告标"低难度"不准确，实际是**中难度**。

### 5.2 工程自评写入 inbox（Claudeception + Homunculus）

报告说"低难度，只改 SK"——**判断基本正确，但低估了真我指出的问题**。

Claudeception 的质量门控有四条硬标准（非显而易见、可复用、有触发条件、已验证）。报告建议我们借鉴，但没设计验收格式。

**数字真我约束**：没有格式定义的 inbox 输入 = 垃圾场。这个机制要做，必须先让数字真我定义验收模板。

### 5.3 SK description 语义优化（Claudeception）

Claudeception README 原文举的例子："Helps with database problems" 太模糊 vs "Fix for PrismaClientKnownRequestError in serverless" 才能被匹配——**报告引用准确**。

技术总工的活，和数字真我无关。**验证通过，无争议。**

### 5.4 本能式渐进学习（Homunculus）

报告说"我们已经在用类似逻辑（积累够了再写入）"——**类比正确**，但标"中难度"偏低。

**数字真我约束**：当前的"3 篇交叉验证"是定性规则，改成数值评分 = 改入库审核标准 = 数据宪法问题。不是工程能决定的。

### 5.5 知识自动召回（semantic-memory + Acontext）

报告标"高难度"——**准确**。Cursor 没有 Claude Code 的 Hook 系统，实现路径完全不同。

**数字真我约束**：自动注入放大了脏数据风险，向量库纯净度要求反而更高。

## 三、报告遗漏

1. **everything-claude-code 被严重低估**。10 万星的项目，包含 skills、instincts、memory、security 四大模块，和我们的体系重叠度可能比报告描述的高得多。技术工程只看了 continuous-learning 子模块就下了结论。
2. **没有评估平台兼容性的统一结论**。6 个项目中 4 个是 Claude Code 专属（Hook 系统），只有 agent-memory-skill 和 Acontext 是跨平台的。报告在每个项目下分别提了"不适用"，但没有在总结中强调：**我们在 Cursor 环境下，能直接用的只有 agent-memory-skill 的理念**。
3. **没有成本估算**。agent-memory-skill 零依赖零成本，Acontext 需要后端服务，semantic-memory 需要 Ollama——报告没有对比运行成本。
