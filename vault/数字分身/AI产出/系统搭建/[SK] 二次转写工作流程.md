## 一、问题定义

### 原始材料特点
- **来源**：会议/访谈录音 → ASR 转写
- **问题**：
  - 口语化严重（"嗯嗯"、"是不是啊"、"你明白没"）
  - 断句错误、标点混乱
  - 方言/语气词残留
  - 发言人切换频繁，语境碎片化
  - 单文件可能 500-1000+ 行

### 目标输出
- **可读性**：书面语，清晰表达说话人的意图
- **可比对**：保留原文，精炼版与原文逐段对应
- **可检索**：结构化，便于后续入库

---

## 二、核心设计原则

### 1. 分段处理（节约 Token）

```
不要：一次性发送 1000 行原文
要：按时间戳或逻辑单元切分，每次处理 50-100 行
```

**切分规则**：
- 按时间戳间隔切分（如每 5-10 分钟一段）
- 按话题切分（讨论 A 事项 / 讨论 B 事项）
- 每段控制在 **50-100 行 / 1500-3000 字**

### 2. 上下比对格式

采用「原文块 + 精炼块」交替呈现：

```markdown
### 段落 1（00:00:00 - 00:05:00）

#### 原文
> 发言人1：嗯嗯。继续说嘛，做完了继续说，行吧，你做了啥子的会上这坨全部做完了...

#### 精炼
**发言人1** 询问发言人2的工作完成情况，质疑其要求50%分成、只给对方20%的理由。

**发言人2** 解释自己不仅做了前期工作，还负责了后期的报销和标书。
```

### 3. 保留关键信息

精炼时**必须保留**：
- 数字（金额、百分比、人数）
- 人名
- 关键结论/决定
- 承诺/待办事项

**可删除**：
- 语气词（嗯、啊、是不是）
- 重复表达
- 无意义的过渡语

## 三、工作流程

### 步骤 0：预处理（人工或脚本）

1. **标记时间戳边界**：在原文中标注每 5-10 分钟一个段落标记
2. **识别参与者**：列出发言人，尽量替换为真实姓名
3. **粗标主题**：快速浏览，给每个大段落打上话题标签

**输出**：带分段标记的原文

### 步骤 1：分段发送

每次只发送一个段落（50-100 行），使用以下 Prompt：

```markdown
## 任务：二次转写

### 原文段落
[粘贴 50-100 行原文]

### 要求
1. 将口语转为书面语，保持原意
2. 保留所有数字、人名、关键结论
3. 删除语气词、重复表达、无意义过渡
4. 输出格式：

#### 精炼版
[逐段书面化的内容，按发言人组织]

#### 关键信息提取
- **涉及金额/比例**：...
- **涉及人员**：...
- **决定/结论**：...
- **待办事项**：...
```

### 步骤 2：人工比对

收到 AI 输出后：
1. **快速扫描**精炼版是否漏掉关键信息
2. **对照原文**确认数字/人名准确
3. **标记问题**（如有歧义或需保留原文表达）

### 步骤 3：拼接归档

将所有段落的精炼版拼接成完整文档：

```markdown
# [文档标题] - 精炼版

> 原始文件：[输入] xxx.md
> 转写时间：2026-xx-xx
> 参与者：发言人1（xxx）、发言人2（xxx）

---

## 段落 1：[话题标签]（00:00 - 05:00）

### 精炼内容
...

### 关键信息
...

---

## 段落 2：[话题标签]（05:00 - 10:00）

...
```

## 四、Token 估算

| 原文规模 | 分段数 | 每段 Token | 总 Token（输入+输出） |
|---------|-------|-----------|---------------------|
| 500 行 | 5-6 段 | ~2000 | ~20,000 |
| 1000 行 | 10-12 段 | ~2000 | ~40,000 |

**对比全文一次性处理**：
- 1000 行原文 ≈ 30,000 Token 输入
- 加上输出 ≈ 50,000+ Token
- **分段处理可节省 20-30%**，且避免上下文溢出导致的质量下降

## 五、变体场景

### 场景 A：只要精炼版（不比对）

简化 Prompt，只输出精炼内容，不保留原文块。

### 场景 B：带批注的精炼

在精炼版中用 `[注: ...]` 标注需要确认的地方：

```markdown
发言人1 提出分配比例为 64:36 [注：原文数字模糊，需确认]
```

### 场景 C：多人会议 → 按人整理

不按时间线，而是按发言人整理：

```markdown
## 发言人1（老板）的核心观点
...

## 发言人2（薛）的诉求
...
```

## 七、与系统的关系

### 定位
- **上游**：ASR 转写（Whisper / 飞书妙记等）
- **本环节**：二次转写（口语 → 书面语）
- **下游**：入知识库 / 画像更新 / 复盘输入

### 与螺旋的关系
- 二次转写是**材料预处理**，为后续碰撞提供高质量输入
- 转写过程中的关键信息提取，可直接喂给数字真我进行认知更新

## 附录：示例

### 原文（节选）

```
发言人 1  00:00:00
嗯嗯。继续说嘛，做完了继续说，行吧，你做了啥子的会上这坨全部做完了，还是什么全部做完了啊，全部做完了。拿50%啊，好，你给他20%的理由是啥子？

发言人 2  00:00:21
我不仅做了前面，我还做了后面。
```

### 精炼版

```
**发言人1**（老板）询问发言人2的工作完成情况，质疑其分配方案：自己拿50%、只给对方20%。

**发言人2** 解释自己的工作范围不仅包括前期，还包括后期的报销和标书。
```

### 关键信息

- **分配比例**：发言人2 提出 50% vs 20%
- **工作内容**：报销、标书
