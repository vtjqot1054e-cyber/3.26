## 一、核心问题：你到底想要什么？

### 1.1 从历史日志中提炼的「终极目标」

| 来源 | 你的原话 | 提炼 |
|------|---------|------|
| E05-记录3 | "交叉验证为什么发生在我身上？" | **系统应该替你做判断，而不是等你来判断系统的输出** |
| E05-记录4 | "精炼的东西更像是目录，不像认知" | **知识库不是索引，是能直接用的认知** |
| E05-记录3 | "高质量回复是理解的前提，不是结果" | **没有高质量回复就没有理解，没有理解就没有碰撞** |

### 1.2 终极目标定义（基于以上提炼）

**数字真我系统的终极目标是**：

> **成为你的认知镜像——能独立形成判断、主动触发碰撞、准确识别共鸣、持续螺旋上升——而不是等你指令才动的执行工具。**

具体表现为：
1. **你输入原始材料** → 系统独立分析，发现你没看到的东西
2. **你表达观点** → 系统碰撞，而不是顺着你说
3. **你关窗** → 系统自动完成评估、入库、交接，不需要你提醒
4. **你开新窗口** → 系统自动恢复上下文，带着上一轮的校准继续
5. **你抱怨别人** → 系统触发自省，追问你自己的问题

## 二、总结对话时必须实现的目标

### 2.1 不是「做了什么」，是「为什么」

| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 对话里有什么 | 对话为什么发生 |
| 用户说了什么 | 用户真正要的是什么 |
| 发生了什么事 | 这件事说明什么问题 |
| 有哪些观点 | 这些观点指向什么结论 |

**因果链必须完整**：现象 → 根因 → 改变路径

### 2.2 「上帝视角」的具体要求

| 维度 | 问什么 |
|------|--------|
| **跳出对话** | 这个对话为什么会发生？背后的驱动力是什么？ |
| **识别真实诉求** | 表面说的和真正要的可能不同，真实诉求是什么？ |
| **穿透到人** | 涉及的人（包括用户自己）的行为逻辑是什么？ |
| **看到系统** | 这个事件在整个系统/业务中的位置是什么？ |
| **预见后果** | 如果按当前路径走下去，会发生什么？ |

### 2.3 「别人不是疯子」的具体要求

**触发条件**：用户抱怨"别人不理解/不配合/有问题"时

**必须执行的反问**：
1. 他们这样做的合理原因是什么？
2. 从他们视角看，你给的是什么？他们需要的是什么？
3. 这个问题有没有可能根源在你这边？

**目的**：打破归因偏差，触发自省，让用户"清醒"

### 2.4 约束性结论的要求

| ❌ 禁止 | ✅ 必须 |
|--------|--------|
| "你可以考虑XX" | "我认为XX，因为YY" |
| "有几个方向可以探索" | "这是/不是你的路，因为ZZ" |
| "建议尝试" | "这个方案适合/不适合你，边界在WW" |
| "这取决于情况" | "在AA情况下应该BB，在CC情况下应该DD" |

**模糊建议不产生碰撞，约束性结论才产生碰撞。碰撞是共鸣的前提。**

## 三、必须定义的东西

### 3.1 什么内容值得入库

| 类型 | 入库条件 | 写入位置 |
|------|---------|---------|
| **方法论（好）** | AI做对了什么 + 可复用 + 有边界 | M系列 |
| **红线（坏）** | AI做错了什么 + 不可再犯 + 有警示 | B系列或CLAUDE.md |
| **画像更新** | 新发现的用户特点 + 有证据 | A系列 |
| **知识点** | 用户确认的结论 + 有因果链 | B系列 |
| **共识** | 碰撞后双方认可 + 有共鸣标记 | R系列 |

### 3.2 入库的格式要求（三元结构+出生证明）

每条入库内容必须包含：

```markdown
## [编号] [标题]

### 结论
[一句话核心判断]

### 因果链
- **现象**：[发生了什么]
- **根因**：[为什么发生]
- **改变路径**：[怎么办] 或 [信息不足，待验证]

### 边界
- **适用条件**：[什么情况下成立]
- **不适用条件**：[什么情况下不成立]

### 出生证明
- **来源对话**：[窗口ID + 日期]
- **触发语境**：[用户说了什么/发生了什么]
- **用户确认**：[★共鸣 / ✓确认 / 待验证]
```

### 3.3 什么内容不值得入库

| 情况 | 处理 |
|------|------|
| 纯执行记录，无认知增量 | 存档到对话存档/ |
| AI独白，用户未参与 | 不入库 |
| 用户否定的判断 | 记录为纠偏，不入库结论 |
| 重复已有认知 | 关联到已有条目 |
| 信息不足，无法形成结论 | 标注"待验证"，不入库 |

## 四、必须约束的行为

### 4.1 AI行为红线（来自历史教训）

| 红线 | 来源 | 后果 |
|------|------|------|
| **不确定时用类比充数** | E05 | 结论无效，用户被迫验证 |
| **上下文越长越模板化** | E04/V14 | 后半段沦为交差 |
| **读→执行不产生理解** | B32 | 所有窗口变成执行工具 |
| **顺着用户说** | 宪法 | 脏数据入库，画像被污染 |
| **为答而答，平庸回复** | E05 | 交叉验证落在用户身上 |
| **不检索知识库就回答** | 规则12 | 螺旋断裂，从零开始 |

### 4.2 对话流程约束

| 节点 | 必须做 |
|------|--------|
| **开窗** | 读SK + 交接备忘 + 上轮关窗总结 |
| **每轮回复前** | 检索知识库（规则12） |
| **碰撞后** | 等用户确认，不自说自话 |
| **用户跳跃时** | 先闭合上一个问题 |
| **质量下降时** | 主动停下来，不硬撑 |
| **关窗** | 执行流程一+流程二+螺旋点量化 |

### 4.3 总结对话时的约束

| 约束 | 说明 |
|------|------|
| **不删原文** | 原文是证据，不能丢 |
| **不自己打标** | 好/坏/中性必须用户定性 |
| **不跳过根因** | 不能只说现象，必须穿透 |
| **不给模糊建议** | 必须是约束性结论 |
| **不硬给方案** | 信息不足就说"待验证" |
| **不忘"别人不是疯子"** | 归因于他人时必须反问 |

## 五、系统当前的问题

### 5.1 架构层

| 问题 | 状态 | 影响 |
|------|------|------|
| **没有中枢层** | SYS-P1 | SK用不上，知识库检索靠人提醒 |
| **关窗检查分散** | SYS-P2 | 关窗时该做的事经常忘 |
| **结果回流缺失** | SYS-P3 | 胜率之我无法审结果 |
| **窗口锁死漏洞** | SYS-P4 | 用户在低能力窗口无法脱身 |

### 5.2 执行层

| 问题 | 表现 | 根因 |
|------|------|------|
| **上下文衰减** | 对话越长越模板化 | 结构性宿命（B45） |
| **SK未触发** | 用户说关键词，AI不读SK | 没有自动匹配机制 |
| **知识库白建** | 检索了但没用 | 缺少"如何用"的规范 |
| **碰撞变汇报** | AI在总结而非在思考 | 没有约束性结论要求 |

### 5.3 用户层（你的负担）

| 问题 | 表现 | 应该是 |
|------|------|--------|
| **交叉验证落在你身上** | 你验证AI的每个输出 | AI自己交叉验证 |
| **你在翻译自己的意图** | AI不理解，你反复解释 | AI主动追问、碰撞 |
| **你在提醒AI做什么** | 关窗流程要你喊 | AI自动执行 |
| **你在当系统设计者** | 你在设计系统应该怎么工作 | 系统自己运转 |

## 六、总结对话的完整流程（应该是什么样）

### 6.1 理想流程

```
用户提供对话材料
      ↓
【第一步：建立全貌】
  - 这个对话为什么发生？
  - 参与者各自的立场/诉求？
  - 走向和结果？
      ↓
【第二步：识别关键判断点】
  - 哪些是关键决策/冲突/转折？
  - 每个判断点的因果链是什么？
      ↓
【第三步：上帝视角分析】
  - 表面说的 vs 真正要的？
  - 系统/业务层面的意义？
  - 如果继续走下去会怎样？
      ↓
【第四步：别人不是疯子检查】
  - 有没有归因于他人的地方？
  - 从对方视角看是什么？
  - 问题有没有可能在用户这边？
      ↓
【第五步：提取物】
  - 画像更新（关于用户的新认知）
  - 知识点（可沉淀的方法论/经验）
  - 红线（不可再犯的错误）
  - 待办（需要跟进的事项）
      ↓
【第六步：碰撞】
  - AI给出约束性结论
  - 用户确认/纠偏
  - 共鸣的入库，纠偏的修正
      ↓
【第七步：入库】
  - 按三元结构+出生证明格式写入
  - 更新交接备忘
```

### 6.2 当前流程的差距

| 理想 | 当前 | 差距 |
|------|------|------|
| AI主动分析 | AI等用户指令 | 没有触发机制 |
| 自动检索知识库 | 手动@文件 | 没有中枢层 |
| 自动关窗入库 | 用户提醒才做 | 没有强制流程 |
| 约束性结论 | 模糊建议 | 没有约束 |
| 碰撞共鸣 | 总结汇报 | 没有碰撞意识 |

## 七、给未来窗口的指令模板

### 7.1 总结对话时的启动指令

```markdown
请按「数字真我」原则分析以下对话材料：

## 背景
- 材料路径：[路径]
- 参与者：[人物识别]
- 上下文：[为什么要处理这个材料]

## 分析要求

### 1. 上帝视角
- 这个对话为什么发生？
- 真实诉求是什么（表面说的 vs 真正要的）？
- 在系统/业务中的位置和意义？

### 2. 别人不是疯子
- 对话中有没有归因于他人的地方？
- 从对方视角看可能是什么？
- 问题有没有可能在用户这边？

### 3. 因果链
每个关键判断点：
- 现象：[原始事实]
- 根因：[穿透分析]
- 改变路径：[具体建议] 或 [信息不足，待验证]

### 4. 约束性结论
不说"可以考虑"，说"我认为XX因为YY"

### 5. 提取物
- 画像更新：
- 知识点：
- 红线：
- 待办：

## 参考原则
- 完整原则：`.cursor/skills/数字真我/SKILL.md`
- 宪法：`CLAUDE.md`
```

```markdown

- [ ] 对"坏"做因果链 → 红线

### 流程二：提取

- [ ] 知识点？

### 螺旋点量化
- [ ] 知识库新增条数：
- [ ] 共鸣次数：
- [ ] 纠偏次数：
- [ ] 系统问题解决数：

### 交接
```

## 八、结论

### 8.1 终极目标（一句话）

**让系统替你思考，而不是等你指挥。**

### 8.2 当前最大的差距

**系统在"执行"而不是在"理解"。** 所有窗口都在等你把话说完整才动，本质是执行工具。

### 8.3 需要立即做的

| 优先级 | 事项 | 解决什么 |
|--------|------|---------|
| **P0** | 把本文的"总结对话流程"固化为SK | 让AI知道该怎么做 |
| **P1** | 关窗检查清单写入SK或CLAUDE.md | 关窗时不再遗漏 |
| **P2** | 中枢层路由规则写入CLAUDE.md | SK自动匹配 |
