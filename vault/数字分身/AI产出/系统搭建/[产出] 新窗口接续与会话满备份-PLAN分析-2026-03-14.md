# 新窗口接续 & 会话满备份 — PLAN 分析

> 用户需求：1) 新窗口说「我在新窗口」→ SK 读最新 SpecStory 接续，省 token；2) 会话快满时系统帮忙：Open as Editor + 自动存对话导出 + 自动命名

---

## 问题 1：新窗口接续（省 Token）

### 需求拆解

| 要素 | 说明 |
|------|------|
| 触发 | 用户在新窗口说「我在新窗口」或语义相近（如「接上文」「接上」「新话题」） |
| 动作 | SK 去 SpecStory 读**时间最新**的备份 |
| 目的 | 有接续能力，但不载入全量上下文，省 token |

### 技术可行性：✅ 高

- SpecStory 自动保存到 `.specstory/history/`，文件名带时间戳：`YYYY-MM-DD_HH-MMZ-话题.md`
- 可按**文件名**或**修改时间**排序，取最新一条
- 新窗口 = 新 context，AI 只需读 1 个文件即可获得上一轮的「压缩快照」

### 方案选项

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **A. 读最新完整文件** | 直接读最新的 `.specstory/history/*.md` | 实现简单 | 长对话文件本身可能很大，仍然吃 token |
| **B. 读最新 + 摘要** | 读最新文件，只取最后 N 轮或前 1/3+后 1/3 | 可控 token | 需约定摘要策略 |
| **C. 读最新 + 交接备忘** | 读最新 SpecStory + 交接备忘 | 交接备忘已有结构，接续质量高 | 依赖交接备忘是否及时更新 |

### 推荐：B 或 C 结合

1. **触发词**：在 SK-04 或新建 SK 中加入——`「我在新窗口」` / `「接上文」` / `「新话题」`
2. **执行**：  
   - 扫描 `.specstory/history/`，按时间取最新 1 个  
   - 可选：只读**最后 3–5 轮**（`### 👤 用户` / `### 🤖` 块）  
   - 或：读 `交接备忘.md` + 最新 SpecStory 的尾部几轮
3. **输出**：用 1–2 句总结「上一轮做了什么、未完成什么」，再问「这次要处理什么？」

### 实现位置

- 新增 SK 或扩展现有 SK（如 SK-01 跨会话监控）  
- 或写进 CLAUDE.md 启动流程：检测到「我在新窗口」→ 读最新 SpecStory + 交接备忘

## 问题 2：会话快满时的自动备份

### 需求拆解

| 要素 | 说明 |
|------|------|
| 触发 | 会话快满时，系统提醒形成交接备份 |
| 期望 | 1) Open as Editor；2) 自动存到对话导出；3) 按系统规则自动命名 |

### Cursor 能力现状（基于技术调研）

| 能力 | 现状 | 来源 |
|------|------|------|
| **Open Chat in Editor** | 存在，手动：More → Open Chat in Editor | Cursor 文档/论坛 |
| **Export Transcript** | 仅手动，右键/菜单导出；无官方 API | Cursor 论坛 |
| **程序化导出** | 无官方 API；社区方案读 SQLite / workspaceStorage | 非官方、易随版本失效 |
| **SpecStory 自动保存** | 有，自动写入 `.specstory/history/` | SpecStory 文档 |

### 「Open as Editor」的含义

- Cursor：把聊天从侧边栏变成**编辑器标签页**（和代码文件一样）
- 用途：全屏查看、便于复制；不会自动导出
- **无 API**：无法由脚本/AI 自动触发，只能用户手动点

### 方案选项

| 方案 | 做法 | 可行性 | 说明 |
|------|------|--------|------|
| **A. 纯手动流程** | 提醒 → 用户手动 Open as Editor → 手动 Export → 手动存到 对话导出 | ✅ | 现状，无自动化 |
| **B. 利用 SpecStory 已有备份** | 提醒 → 用户确认 → 跑 `export_transcript.py` 处理**最新** SpecStory 文件 → 写入 对话导出 | ✅ | SpecStory 已自动存，只需再清洗+命名 |
| **C. Cursor Automations** | 若 Cursor 有 Automations，尝试配置「定时/事件 → 导出」 | ⚠️ 待验证 | Automations 偏 CI/CD，未必支持聊天导出 |
| **D. 读本地 DB** | 脚本读 workspaceStorage / SQLite 取当前对话 | ⚠️ 脆弱 | 无文档、格式可能变 |

### 推荐：B（SpecStory + export_transcript.py）

**流程**：

1. **会话快满时**：AI 提醒「请形成交接备份」
2. **用户确认**：回复「存」或「备份」等
3. **AI 执行**：  
   - 找到 `.specstory/history/` 中**最新**文件（当前窗口应对应最新或次新）  
   - 调用 `export_transcript.py` 清洗  
   - 输出到 `数字分身/对话导出/`，命名：`[导出] YYYY-MM-DD-HHMM-话题.md`

**依赖**：SpecStory 已自动保存当前对话。若用户在提醒后**立即关窗**，有可能尚未写入，可加一步：先执行 `SpecStory: Save AI Chat History`（Command Palette）再跑脚本。

### 「Open as Editor」的落位

- 无法自动化，但可写进**用户操作清单**：  
  「交接前建议：More → Open Chat in Editor（便于全屏检查）→ 确认后关窗 → SpecStory 已自动存」
- 若 Cursor 后续提供 API，再考虑自动化。

---

## 实施顺序建议

| 阶段 | 内容 | 优先级 |
|------|------|--------|
| 1 | 新窗口接续：触发词 + 读最新 SpecStory（含尾部 N 轮或交接备忘） | P0 |
| 2 | 会话满备份：提醒话术 + 调用 export_transcript 处理最新 SpecStory | P0 |
| 3 | 在 CLAUDE/交接备忘中写明「Open as Editor」为可选人工步骤 | P1 |
| 4 | 跟踪 Cursor 官方导出 API / Automations 更新 | P2 |

---

## 约束与风险

- SpecStory 写入时机：关窗或手动 Save 时写入，若用户未关窗就提醒，可能拿到的是上一窗口的「最新」
- 多窗口：多个对话并行时，「最新」未必是当前窗口，需在提醒中说明以「当前窗口对应的文件」为准
- Cursor 无导出 API：所有自动化都依赖 SpecStory 本地文件，SpecStory 为事实上的数据源
