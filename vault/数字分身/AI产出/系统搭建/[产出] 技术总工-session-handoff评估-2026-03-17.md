### 功能分析

| 它做什么 | 我们现在怎么做 | 差距在哪 |
|----------|--------------|----------|
| **触发条件覆盖6种场景**：(1)用户请求 (2)上下文快满 (3)里程碑完成 (4)工作结束 (5)特定口令 (6)恢复续接 | 只覆盖4种：用户口令、上下文快满、任务完成、大脑指令关闭 | **缺少"主动建议"机制**——session-handoff在5+文件编辑/复杂调试/架构决策后会主动提醒 |

| **过期检测**：check_staleness.py检查交接文档时效性（FRESH/SLIGHTLY_STALE/STALE/VERY_STALE） | 无过期概念——交接备忘只有最新状态 | **不严重**——我们的交接备忘是实时覆盖式的，不是文档堆叠式的 |
| **链式交接**：--continues-from 参数链接前序文档 | 窗口命名+SpecStory自动存档 | **不需要**——我们的架构已解决续窗口问题 |
| **模板结构**：12个必填节（架构概览、决策理由、关键文件、潜在陷阱、环境状态等） | 三向交付模板（上级/续窗口/关联窗口） | **我们的三向交付更适合多窗口体系**，但可以从模板中借鉴"决策理由""潜在陷阱"字段 |

### 引入评估

| 维度 | 判断 |
|------|------|
| 与现有体系兼容性 | **低** — session-handoff是单窗口设计（一个Agent一个handoff），我们是多窗口多层级体系（总大脑/分大脑/执行窗口各有不同交付需求）。Python脚手架在Cursor中不适用 |
| 引入收益 | **低** — 模板结构、脚本自动化对我们价值不大。真正有价值的是**触发机制设计思路** |
| 引入成本 | **中** — 安装容易（npx一键），但适配成本高：需要改造三向交付结构、解决多窗口冲突、处理Python脚本依赖 |
| 替代方案 | **借鉴触发机制+自检机制**，改进我们自己的交接协议SK |

### 结论

- **建议：不引入，借鉴核心思路**（与总大脑预判一致）
- **借鉴什么**：三个具体机制（详见下方1.2节）
- **改进哪个SK**：`.cursor/skills/00-通用-交接协议/SKILL.md`
- **需要谁执行**：架构师/工程修改SK

## 1.2 session-handoff 可移植的触发机制（重点分析）

从session-handoff的完整SK中提炼出三个值得移植的机制：

### 机制A：主动建议触发（Proactive Suggestion）

**session-handoff原文**：
> After substantial work (5+ file edits, complex debugging, major decisions), suggest:
> "We've made significant progress. Consider creating a handoff document..."

**我们的问题（CLAUDE.md核心矛盾第3条）**：
> 上帝视角还是呼唤制——AI没有主动打断的肌肉记忆，等技术突破

**可移植性分析**：

session-handoff用"5+文件编辑"作为量化阈值。这个思路在我们的体系中需要转化——Cursor窗口不方便计数文件编辑，但可以用**对话轮次 + 内容类型**作为等效触发条件：

| 触发指标 | 阈值 | 理由 |
|----------|------|------|
| 对话轮次 | ≥20轮 | 20轮约消耗上下文的40-60%，再不提醒就可能来不及 |
| 产出了架构决策 | 任何一个 | 架构决策必须落盘，不能丢在对话里 |
| 产出了新文件 | ≥3个 | 多文件修改=大量上下文占用 |
| 用户情绪转折 | 检测到 | 情绪转折是高价值信号，必须及时记录 |
| 话题跳跃 | ≥2次 | 话题跳跃说明窗口职能可能模糊，提醒评估是否该开新窗口 |

**移植形式**：在交接协议SK的"触发条件"中新增"AI主动建议"节，给出具体话术模板和量化阈值。

### 机制B：交付自检清单（Validation）

**session-handoff原文**：
> The validator checks:
> - No TODO placeholders remaining
> - Required sections present and populated
> - No potential secrets detected
> - Quality score (minimum 70)
> - Referenced files exist

**我们的问题**：
当前交接协议只规定了"写什么"，没有规定"写完后检查什么"。AI可以敷衍填充，没人验证。

**可移植性分析**：

我们不需要Python脚本验证，但可以把自检逻辑**内化到SK的约束中**——AI在完成交接后，必须逐条自检：

| 自检项 | 规则 |
|--------|------|
| 产出文件列表是否完整 | 本轮新建/修改的文件必须全部列出，不能漏 |
| 断点描述是否具体 | "做了一半"不合格，必须说"做到XX文件的第XX步" |
| 未完成项是否标原因 | 每个未完成项必须附原因+建议（延后/取消/转派） |
| 是否包含决策理由 | 如果本轮做了选择/判断，必须写清楚为什么这样选 |
| 上下文摘要≥3条 | 少于3条说明摘要不充分 |

**移植形式**：在交接协议SK的"交付后的文件操作"之后新增"交付自检"节。

### 机制C：预填信息思路（Smart Scaffold）

**session-handoff原文**：
> The script will pre-fill: timestamp, project path, git branch, recent commits, modified files

**我们的问题**：
交接备忘中的"时间""窗口名称""角色"等字段每次都要手写。

**可移植性分析**：

虽然不用Python脚本，但可以在SK中**规定AI必须自动填充的字段**（这些信息AI在对话上下文中都有）：

| 自动填充字段 | 来源 |
|-------------|------|
| 窗口名称 | 窗口自述中已声明 |
| 角色/层级 | 窗口自述中已声明 |
| 开始时间 | 窗口自述中已声明 |
| 结束时间 | 当前系统时间 |
| 本轮产出文件列表 | AI回溯对话中的文件操作 |

**移植形式**：在交付模板中标注哪些字段是"AI自动填充"，哪些是"需要AI思考后填写"。

## 1.3 交接协议SK改进方案（具体到哪段改什么）

### 改动1：触发条件新增"AI主动建议"

**当前内容**（交接协议SK 第10-16行）：

```markdown
## 触发条件

- 用户说"关窗口"、"结束"、"交付"、"交接"
- 上下文接近容量极限（AI 应主动提醒）
- 任务全部完成，准备关闭
- 大脑指令"该窗口可以关了"
```

**改为**：

```markdown
## 触发条件

### 被动触发（用户/上级发起）
- 用户说"关窗口"、"结束"、"交付"、"交接"
- 大脑指令"该窗口可以关了"

### 主动触发（AI自行判断，强制）

AI在对话过程中必须持续监控以下指标，**任一达到阈值时主动提醒用户**：

| 指标 | 阈值 | 提醒话术 |
|------|------|----------|
| 对话轮次 | ≥20轮 | "已经20轮了，建议做一次中间落盘，避免信息丢失。需要现在交接吗？" |
| 上下文接近满 | 感知到回复变慢/内容丢失 | "上下文快满了，必须现在交接。我先写交接备忘。" |
| 架构决策产出 | 任何一个 | "刚才做了架构决策，建议立刻落盘到文件，不等对话结束。" |
| 新建/修改文件数 | ≥3个 | "本轮已修改3+文件，建议做中间交接，防止后续上下文丢失这些信息。" |
| 话题跳跃 | ≥2次大跳 | "话题跳了两次了，当前窗口的职能边界可能模糊——建议先交接当前任务，再评估是否开新窗口。" |

**"上下文快满"的主动触发是最高优先级——不需要等用户同意，直接进入交接模式。**
其他主动触发需要用户确认后才执行。

### 特殊触发：中间落盘（不关窗口）
- 触发条件：架构决策产出 / 文件修改≥3个 / 用户情绪转折
- 行为：**不触发完整三向交付**，只把关键结论/决策写入对应文件
- 目的：防止"对话里说了但没写进文件"的信息丢失
```

### 改动2：交付模板增加"决策理由"和"潜在陷阱"

**当前内容**（交接协议SK 第49-72行的交付模板）：
在 `### 续窗口交付` 下方增加两个字段。

**在模板的 `### 续窗口交付` 部分改为**：

```markdown
### 续窗口交付
- **断点**：[做到哪了——必须具体到文件名+步骤，"做了一半"不合格]
- **未完成**：[具体待办]
- **上下文**：[下个窗口必须知道的3-5条关键信息]
- **本轮决策及理由**：[做了什么选择、为什么这样选——没有决策写"无"]
- **潜在陷阱**：[下个窗口容易踩的坑——没有写"无"]
```

### 改动3：新增"交付自检"节

**在当前"交付后的文件操作"（第89-93行）之后新增**：

```markdown
## 交付自检（AI完成交接后必须逐条核对）

| # | 检查项 | 合格标准 | 不合格处理 |
|---|--------|---------|-----------|
| 1 | 产出文件列表完整 | 本轮新建/修改的文件全部列出 | 回溯对话补全 |
| 2 | 断点描述具体 | 包含文件名+步骤/位置 | 重写，不允许"做了一半" |
| 3 | 未完成项标原因 | 每项附原因+建议（延后/取消/转派） | 逐条补充 |
| 4 | 决策理由已记录 | 本轮每个选择/判断都有"为什么" | 补写理由 |
| 5 | 上下文摘要≥3条 | 少于3条=摘要不充分 | 补充到≥3条 |
| 6 | 关联窗口影响已评估 | 即使"无影响"也要显式写出 | 补写"无" |

**自检不通过不能关窗口。** 先补全再关。
```

### 改动4：在约束中强化"中间落盘"原则

**在当前"约束"（第95-99行）中新增一条**：

```markdown
- **重要结论必须即时落盘**——不等关窗口。架构决策、方向修正、关键发现产出时立刻写入文件。对话关了就没了，文件才能跨窗口传递（继承CLAUDE.md"即时落盘"原则）
```

# 二、OB标记方式评估

## 2.1 用户痛点

用户习惯Word式操作：选中→右键批注→写想法→高亮标色。当前OB的markdown标记对用户反直觉：

| 当前标记 | 用户体验问题 |
|----------|-------------|
| `==高亮==` | 需要手打符号，不能用快捷键选中高亮 |
| `~~删除线~~` | 手打符号繁琐 |
| 脚注`[^N]` | 需要在文末补充脚注内容，断开注意力 |
| 直接插入文字评论 | 和原文混在一起，下游难区分 |

## 2.2 候选方案调研

### 方案A：CriticMarkup 插件（obsidian-criticmarkup）

| 项 | 内容 |
|---|------|
| 工具名 | obsidian-criticmarkup |
| 来源 | GitHub: Fevol/obsidian-criticmarkup |
| 查证状态 | ✅ 已查证 |
| Stars | 218 |
| 最后更新 | 2025年12月（v0.2.6） |
| 兼容平台 | Obsidian（Live Preview + Reading View） |

**核心功能**：

| 功能 | 语法 | 用户体验 |
|------|------|----------|
| 删除标记 | `{--text--}` | 选中→命令面板/右键→标记删除 |
| 添加标记 | `{++text++}` | 选中→标记添加 |
| 替换标记 | `{~~old~>new~~}` | 选中→标记替换 |
| 高亮 | `{==text==}` | 选中→标记高亮 |
| **批注（关键功能）** | `{>>comment<<}` | **右侧边栏显示批注，hover可看** |
| 接受/拒绝 | 命令面板/右键 | 类似Word的接受/拒绝修订 |
| 作者+时间戳 | 自动元数据 | 建议模式自动添加 |
| 全库索引 | 自定义视图 | 可以看所有文件的批注/建议 |

**用户体验评级**：⭐⭐⭐⭐ — 最接近Word批注体验。选中文字→右键或快捷键→添加批注/高亮/删除标记。批注在右侧边栏显示，不污染正文。

**风险**：
- 🔴 **Beta状态，官方警告"不要用在主库"** — 有潜在数据丢失风险
- 🔴 语法和标准Markdown不同（`{>><<}` vs `==`），管线SK需要改造

**核心功能**：

| 功能 | 说明 |

| 高亮+hover | 标记文字高亮，悬浮显示旁注 |
| 侧边栏面板 | 显示当前文件/全库旁注 |
| AI功能 | 需要Ollama，找相关内容（我们不需要） |

**用户体验评级**：⭐⭐⭐ — 右键添加旁注的体验接近Word，但功能单一（只有旁注，没有删除/替换标记），且项目极新、星数极低。

**风险**：
- 🔴 **只有4个Star，项目成熟度极低**
- 🔴 作者自称"vibe-coded"，代码质量存疑
- 🟡 不支持删除标记和替换标记，只有旁注功能

| 项 | 内容 |
|---|------|

**核心功能**：
- 行内评论工具提示

**用户体验评级**：⭐⭐ — 功能简单，但维护状态不确定，功能不如CriticMarkup丰富。

### 方案D：Highlights Sidebar 插件

| 项 | 内容 |
|---|------|

- 点击可跳转、自动刷新

**用户体验评级**：⭐⭐⭐ — 不改变标记方式，但提供了更好的**审阅视图**。和现有标注体系完全兼容。

### 方案E：维持现状 + 快捷键优化

不安装新插件，通过OB内置快捷键/Templater优化体验：
- 设置 `Ctrl+H` = 高亮（`==选中文字==`）
- 设置 `Ctrl+D` = 删除线（`~~选中文字~~`）
- 脚注用Templater模板快速插入

**用户体验评级**：⭐⭐ — 降低手打符号的痛苦，但本质没变，仍然不是Word级体验。

## 2.4 推荐方案

### 短期（立即执行）：方案D + 方案E 组合

**理由**：
1. 管线零改动——当前阶段的核心矛盾是"把地基打好"（CLAUDE.md），不应为了标记体验而大改管线

3. 快捷键优化降低手打符号的摩擦
4. **风险为零**——标准OB语法，不引入新依赖

**安装方式**：
```

3. OB设置 → 快捷键 → 搜索 "Toggle strikethrough" → 设为 Ctrl+D
```

**管线适配改动**：无。

### 中期（管线稳定后再评估）：方案A CriticMarkup

**理由**：
1. CriticMarkup是最接近Word批注体验的方案，用户痛点解决最彻底

- [ ] 管线已跑通≥10篇对话的完整流程

- [ ] 在测试库中验证CriticMarkup语法不影响WF1/WF2/WF3
