# 中枢层架构与对话入库方案

> 生成时间：2026-04-03
> 背景：用户痛点——无中枢层消化指令、SK用不上、对话无法自动入库

---

## 一、问题诊断

### 你的三个痛点

| 痛点 | 根因分析 |
|------|---------|
| **1. 没有中枢层消化指令** | 用户输入直接进入窗口，没有「路由+预处理」层匹配 SK/知识库 |
| **2. 无数 SK 用不上** | SK 是被动的（需要用户记得用），不是主动匹配的 |
| **3. 对话无法自动入库** | 关窗时没有「评估→确认→入库」的标准流程 |

### 核心缺失

```
现状：
用户输入 → 直接到窗口 → 窗口从零开始 → 无知识库支撑

期望：
用户输入 → 【中枢层】→ 匹配SK + 检索知识库 + 注入上下文 → 窗口
                    ↑
                    └── 关窗时：对话评估 → 入库
```

## 二、技术调研结果

### 2.1 对话历史导出：cursor-history-mcp

**已有方案**：[cursor-history-mcp](https://github.com/s2thend/cursor-history-mcp)

| 能力 | 说明 |
|------|------|
| `cursor_history_list` | 列出所有会话 |
| `cursor_history_show` | 查看完整对话内容 |
| `cursor_history_search` | 跨会话搜索 |
| `cursor_history_export` | 导出为 **Markdown 或 JSON** |
| `cursor_history_backup` | 完整备份 |

**安装**：
```bash
npx cursor-history-mcp
```

**与你的需求匹配**：
- ✅ 可以导出对话为 Markdown → 直接放入 OB vault
- ⚠️ 但**没有入库评估逻辑**——需要你自己加

---

### 2.2 中枢层/路由层：2026 年主流架构

**核心模式**：Signal-Decision Architecture

```
用户输入
    ↓
【信号提取】→ 意图、紧急度、复杂度、领域
    ↓
【决策引擎】→ 选择哪个 SK、加载哪些知识库条目
    ↓
【注入上下文】→ 系统提示 + 知识片段 + 画像
    ↓
窗口执行
```

**关键技术**：
1. **混合路由**：80% 用正则/关键词匹配（免费、即时），20% 用 LLM 分类（需要推理）
2. **语义缓存**：相似问题直接命中缓存，避免重复检索
3. **渐进式加载**：先摘要，深入时再加载全文

**共同模式**：
```
对话中 → store_memory（存储）
下一轮 → recall_memories（召回）
```

### 3.2 分阶段实施

#### 阶段 A：对话入库流程（最痛 → 先解决）

**目标**：关窗前，对话能被评估并入库

**方案**：
```
1. 安装 cursor-history-mcp
   npx cursor-history-mcp

2. 关窗时手动触发（或写脚本）：
   - cursor_history_export → 导出当前会话为 Markdown
   - 放入 OB: vault\数字分身\对话导出\

3. 在 Cursor 中开新窗口：
   - 读取刚导出的 Markdown
   - 用「复盘大脑 SK」评估：
     - 有无方法论/红线？
     - 有无画像更新？
     - 有无共识/知识点？
   - 用户确认后，写入 knowledge_base_v2
```

**入库标准（简化版）**：
```markdown
## 入库评估清单

### 1. 方法论/红线（流程一产出）
- [ ] 这轮对话有没有 AI 做对/做错的事？
- [ ] 能否提炼为可复用的规则？

### 2. 画像/知识点/共识（流程二产出）
- [ ] 有没有新发现的用户特点？
- [ ] 有没有用户确认的结论？
- [ ] 有没有可入库的知识点？

### 3. 判断
- [ ] 值得入库 → 写入 knowledge_base_v2
- [ ] 不值得入库 → 存档到 对话导出/ 备查
```

---

#### 阶段 B：中枢层 MVP（SK 自动匹配）

**目标**：用户输入时，自动匹配相关 SK 和知识库

**方案**：在 CLAUDE.md 中加入「路由规则」

```markdown
## 中枢层路由规则（规则14）

### 意图识别（正则优先）
| 关键词 | 匹配 SK | 知识库检索 |
|--------|--------|-----------|
| 复盘、总结、关窗 | 复盘大脑 | M系列方法论 |
| 启动数字真我 | 数字真我 | 全局画像 |
| 投标、标书 | 投标顾问/投标工程 | B系列知识 |
| 二次转写、口语 | - | 二次转写方案 |
| 螺旋、原则、架构 | 数字真我 | 系统搭建原则 |

### 执行流程
1. 识别用户输入中的关键词
2. 匹配 SK → 读取对应 SKILL.md
3. 检索 Chroma → 加载相关条目
4. 组装上下文后回复
```

**技术实现**：
- 短期：写入 CLAUDE.md，靠模型自觉遵守
- 中期：用 MCP 工具实现自动检索（chroma_query）
- 长期：独立中枢层服务（脚本/API）

---

#### 阶段 C：持久化记忆（跨窗口延续）

**目标**：下一个窗口能"记得"上一轮的校准结果

**方案**：
```
1. 交接备忘.md 继续作为「当前系统状态」的单一真相源
2. 每次关窗时更新交接备忘：
   - 本轮共鸣/纠偏
   - 螺旋点进展
   - 待办事项
3. 新窗口启动时：
   - 读 SK
   - 读交接备忘
   - 读上一轮关窗总结
```

**可选增强**：
- 接入 Memori MCP / Remembra MCP
- 用 `store_memory` 存储关键共识
- 新窗口启动时 `recall_memories` 召回

## 六、与螺旋的关系

这个中枢层设计本身就是**螺旋的一部分**：

```
用户输入
    ↓
中枢层检索知识库 ← 螺旋锚点（规则12）
    ↓
窗口执行 + 碰撞
    ↓
关窗入库评估
    ↓
更新知识库 → 下一轮中枢层检索更准
    ↓
（螺旋上升）
```

**没有中枢层 = 螺旋断裂**：每次从零开始，知识库白建。
