## 一、背景

用户向工程窗口展示两类新信息：

1. **Obsidian CLI**（官方文档/视频）：命令行控制 Obsidian 应用，宣称「在 Obsidian 里能做的均可通过 CLI 做」，含 `append`、`backlinks`、`command`、`base:query` 等与 Bases/书签/命令执行相关的子能力；侧边栏与 Obsidian Headless、URI 并列，方向是「脚本化、自动化、与外部工具集成」。
2. **Supermemory**（商业/开源产品宣传）：自称「AI 记忆与上下文引擎」，强调从对话中抽取事实、用户画像、知识更新与矛盾处理，并引用 LongMemEval / LoCoMo / ConvoMem 等记忆评测语境。

本报告将二者对照当前**家里已落地**的通路：`knowledge_base_v2` + Smart Connections + ChromaDB + Cursor MCP（INFRA-003/004 已完成），供真我判断是否与「对话驱动认知积累」「管线入库」「SYS-001/SYS-002」等长期目标对齐。

## 二、Obsidian CLI — 要点与启发

### 2.1 本质（可验证事实）

- Obsidian 提供 CLI，用于在终端调用与 Obsidian 应用协同的能力（含开发向能力如插件重载、截图等，以官方文档为准）。
- 与 **Obsidian URI**、**Headless** 同属「扩展 Obsidian」条目，说明官方在推进「无界面 / 低界面」下的仓库操作。

### 2.2 与当前架构的关系（推断）

| 现状 | CLI 可能补位 |
|------|----------------|
| AI 产出多经文件系统写入 vault | 可改为显式 `append`/命令式写入，减少路径与编码误操作 |
| 双链、书签、Bases 多为人工或半自动 | 可脚本化查询 `base:query`、`backlinks` 等（以实际命令为准） |
| Smart Connections 索引依赖 OB 内行为 | CLI 若可触发命令或保存流程，或能缩短「入库 → 可被 OB 插件感知」的闭环（需实测） |
| N8N / 外部自动化 | 可增加「调用 Obsidian CLI」一步，与现有文件监听并列 |

### 2.3 启发（供真我判断）

1. **「读」已由 MCP + Chroma 部分承担；「写」若长期靠纯文件写入，与 OB 语义层（双链、书签、命令）可能脱节。** CLI 是「写回 OB 语义世界」的官方通道候选。
2. **若 WF1/WF3 或数字真我入库希望与 Obsidian 内部状态一致**（而非仅磁盘上有 .md），CLI 值得列入「集成选项」而非必选。
3. **风险**：依赖 Obsidian 本体是否运行、版本与插件兼容性、Headless/CLI 的实际限制需**按官方文档与本地实测**再下结论；工程窗口未在本机执行安装与验收。

### 2.4 工程侧建议（非决议）

- **短期**：不改动 INFRA 已验收链路；将 CLI 记为 **ARCH/INFRA 候选任务**（例如「Obsidian CLI PoC：单 vault 读写 + 与 knowledge_base_v2 对齐」）。
- **中期**：若真我批准，可由工程做**最小 PoC**（读一篇笔记、追加段落、列 backlinks），再交给管线/真我定义是否替代部分文件写入。

## 三、Supermemory — 要点与启发

### 2.1 本质（需核实）

- 产品定位：**会话级长期记忆 + 上下文编排**，强调自动抽取、画像、更新与矛盾处理；具体实现、开源程度、自建/云服务、数据出境与成本需查官方仓库与条款（本报告仅凭用户提供的宣传材料，**不构成产品认证**）。

### 2.2 与 knowledge_base_v2 + Chroma 的差异（用户体系内对比）

| 维度 | 当前 knowledge_base_v2 + Chroma | Supermemory 类系统（宣传口径） |
|------|-----------------------------------|--------------------------------|
| 内容形态 | 精炼 ABC、因果链、审校门槛高 | 偏「从对话自动生长」的记忆单元 |
| 质量闸门 | 数字真我 / 胜率之我 / 用户 | 产品内置抽取与冲突策略（黑盒程度需评估） |
| 时间演化 | 依赖人工改版、标签与归档 | 宣传强调时间线与矛盾处理 |
| 检索 | Chroma 向量 + OB Smart Connections | 产品自研记忆层（未必与现有向量库同构） |

### 3.3 启发（供真我判断）

1. **与宪法「不顺图用户、入库需审」的张力**：全自动记忆若绕过审校，可能放大**脏数据**风险；若只做「候选记忆 → 待确认 → 真我入库」，则与现有 WF1→待确认→复盘管线**可同构**。
2. **与 B35/V16「因果链 + 图谱」方向**：Supermemory 擅长**事件与事实的时序与会话绑定**；精炼库擅长**可辩论结论**。二者或可分层，而非二选一。
3. **与 SYS-002（结果回流）**：若未来要有「预判 → 结果 → 回写知识库」，会话记忆层可能是**中间态**，但替代不了精炼层。

### 3.4 风险（必须写清）

- 供应商锁定、隐私、合规、与本地 Chroma 双用时的**一致性问题**。
- 评测名次与真实业务场景（中文、长对话、政治业务语境）是否匹配，需**独立验证**，不可直接作为架构依据。

### 3.5 工程侧建议（非决议）

- 记为 **「待真我裁决的技术侦查项」**；在「MCP + 本地 Chroma 尚未充分用起来」之前，**不建议**并行启动重集成。
- 若真我批准侦查：仅做 **read-only 调研清单**（许可协议、API、是否自建、与 MCP 兼容性），产出不超过一页的 GO/NO-GO。
