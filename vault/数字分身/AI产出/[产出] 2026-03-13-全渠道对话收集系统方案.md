# 全渠道对话收集系统方案

> 数字分身"感知神经系统"：多源对话自动采集 → 清洗 → 结构化 → 审核入库
> 讨论来源：2026-03-13 Claude + Gemini 多轮推演
> 前置方案：[[产出] 2026-03-12-清洗车间架构方案]]

---

## 一、定位

**不管 Brain 通过什么界面和外界交互，所有对话都自动汇入同一条管道，成为清洗车间的原料。**

与清洗车间的关系：
- 清洗车间（2026-03-12方案）= 通用内容清洗引擎
- 本系统 = 清洗车间的 **AI对话** 数据源管道
- 其他数据源（钉钉转写、网络视频等）走原料收件箱，不经过本系统

---

## 二、四层架构

```
═══════════════════════════════════════════════════════════
 第一层：采集（哑管道，只搬不洗）
═══════════════════════════════════════════════════════════

 Cursor采集器 ─→ 原样追加 ─→ brain/raw/[原始] YYYY-MM-DD-{device}.md
 Gemini采集器 ─→ 原样追加 ─→     （同目录，按设备分文件）
 (未来更多源)  ─→ 原样追加 ─→

 ▸ 采集器只管搬运，不做任何清洗判断
 ▸ 原始文件包含英文、图片路径、<think>块，全不管
 ▸ 写完后 ping N8N Webhook（5分钟无新写入触发）
 ▸ 原始文件在 Obsidian 中排除索引（设置→排除文件夹）

═══════════════════════════════════════════════════════════
 第二层：清洗（N8N 工作流，自动）
═══════════════════════════════════════════════════════════

 N8N 工作流（触发：Webhook 或定时）
 ├── 节点1 [Code]: 轻清洗
 │     去XML标签、图片路径、ANSI码、<think>块、纯英文段落
 ├── 节点2 [Code]: 实体标准化（读全局实体字典）
 ├── 节点3 [HTTP]: LLM 结构化提取（话题切段、事实/观点/待验证）
 ├── 节点4 [Code]: 组装标准MD + YAML frontmatter (status: pending)
 └── 节点5 [Git]:  push 到 碎片暂存/

 ▸ 所有清洗脚本都在 N8N Code 节点里，不另建 Python 文件
 ▸ N8N 绝对规则：只新建文件，绝不覆写 pending/reviewing/done 文件

═══════════════════════════════════════════════════════════
 第三层：审核（Obsidian，人工）
═══════════════════════════════════════════════════════════

 审核台仪表盘（Dataview 查询页）
 ├── 🔴 审核中（status: reviewing）—— 被打断的优先
 ├── 🟡 待审核（status: pending）—— 新产出
 └── 🟢 已完成（status: done）—— 最近10条

 ▸ 审核 = 改 frontmatter status 字段，不移动文件
 ▸ 中断时在当前位置打 > 🔖 审到这里，改 status: reviewing
 ▸ 发现大面积错误 → status: rejected → N8N 检测到后重新清洗

═══════════════════════════════════════════════════════════
 第四层：查看（对话监控 GUI / Obsidian）
═══════════════════════════════════════════════════════════

 对话监控 GUI：
 ├── 读取 brain/raw/ 原始文件
 ├── 渲染时戴"滤镜"（跳过 <think>、图片路径）
 └── 不写文件、不做清洗，只是实时显示

 Obsidian：
 └── 直接浏览 对话存档/ 或 碎片暂存/
```

## 四、消息格式

### 原始格式（第一层写入）

```markdown
# 对话原始记录 2026-03-13 (office)

---

### 👤 用户 [14:30:25] via Cursor

（原始全文，可能包含图片路径、英文、任何内容）

---

### 🤖 Claude [claude-4.6-sonnet] [14:30:45] via Cursor

（原始全文，可能包含 <think> 块）

---

### 🤖 Gemini [gemini-2.5-pro] [15:10:30] via Gemini

（原始全文）
```

**关键设计决策：**
- 不使用全局递增编号（分布式下必冲突），用时间戳排序
- 每条消息带 `via 来源` 标记
- 文件按日期+设备分片，消灭 Git 冲突
- 设备标识存在 `brain/device_id.txt`，首次运行配置

### 清洗后格式（第二层 N8N 产出）

```markdown
---
type: 结构化碎片
source: Cursor对话 / Gemini对话
date: 2026-03-13
device: office
entities: ["张兰心", "北京公司"]
status: pending
---

## 事实
（剥离情绪的客观骨架）

## 原话/观点
（保留关键表达和逻辑链）

## 待验证
（信息不足、指代不明的部分）

## 原文对照
（折叠，保留完整原文供回查）
```

## 五、Git 同步与冲突防范

### 策略：按设备分文件

```
brain/raw/
├── [原始] 2026-03-13-office.md    ← 公司电脑写
├── [原始] 2026-03-13-home.md      ← 家里电脑写
└── [原始] 2026-03-14-office.md    ← 不同日期

用户输入/碎片暂存/
├── [碎片] 2026-03-13-143025.md    ← N8N 产出，文件名带时间戳
├── [碎片] 2026-03-13-151030.md
└── ...
```

- 不同设备写不同文件 → 无 Git 冲突
- N8N 产出文件名带时间戳 → 不会重名
- 审核修改的是已有文件的 frontmatter → 单向修改，不会冲突

### 工作流

```
到公司 → git pull → 采集器自动启动 → 写 raw/*-office.md
下班前 → git push
回家   → git pull → 采集器自动启动 → 写 raw/*-home.md
睡前   → git push
```

## 六、N8N 触发机制

### 推拉结合（采纳 Gemini 建议）

| 方式 | 触发条件 | 场景 |
|------|---------|------|
| **Webhook 推送** | 采集器检测到 5 分钟无新写入 → ping N8N | 对话刚结束，尽快清洗 |
| **定时轮询** | 每 4 小时扫一次 raw/ | 兜底，防止 Webhook 漏发 |

### N8N 安全规则

- ✅ 只新建文件到 碎片暂存/
- ❌ 绝不覆写任何已存在的文件
- ❌ 绝不修改 status 为 reviewing 或 done 的文件
- ✅ 检测到 status: rejected → 重新清洗该原始数据

## 七、Obsidian 审核流程

### 所需插件

| 插件 | 作用 | 必要性 |
|------|------|--------|
| **Dataview** | 审核仪表盘查询 | 必装 |
| **Meta Bind**（可选） | 一键切换 status 按钮 | 舒适性提升 |
| **Kanban**（可选） | 看板拖拽式审核 | 视觉型审核者 |

### 审核台仪表盘

新建 `数字分身/审核台.md`，内容使用 Dataview 查询：
- 🔴 审核中（reviewing）排最前
- 🟡 待审核（pending）按日期倒序
- 🟢 已完成（done）最近10条

### 状态机

```
N8N产出 → pending → reviewing → done
                       ↑          │
                       └──────────┘ （发现新问题可回退）

                  pending → rejected → N8N重新清洗 → pending
```

### 断点标记

被打断时：
1. 在当前位置打 `> 🔖 审到这里`
2. 改 frontmatter `status: reviewing`
3. 回来后：审核台最顶部 → 打开 → Ctrl+F 搜 🔖 → 继续

## 八、场景矩阵

| 场景 | 操作流程 |
|------|---------|
| **正常审核** | 审核台 → 打开 pending → 逐段读 → 有错直接改 → status: done |
| **审到一半被打断** | 打 🔖 → status: reviewing → 走人 → 回来搜🔖继续 |
| **发现 LLM 提取错误** | 展开原文对照 → 直接改内容 → done（小错）/ rejected（大错） |
| **积了一堆** | 快速扫描：30秒/个，一眼过=done，有疑=reviewing，明显错=rejected |
| **审核中 N8N 产出新文件** | 不影响当前文件（N8N 只新建不覆写）→ 审完刷新看新增 |
| **跨设备继续** | git push → 换电脑 git pull → 审核台看 reviewing → 搜🔖继续 |

## 十、与清洗车间的关系

```
本系统（对话收集管道）          清洗车间（2026-03-12方案）
──────────────────         ──────────────────────
brain/raw/ 原始对话    →     N8N 工作流（共用基础设施）
                              ↕
原料收件箱/ 其他素材    →     N8N 工作流（另一个触发器）
                              ↓
                           碎片暂存/（统一产出）
                              ↓
                           Obsidian 审核台（统一审核）
```

两个系统共用：N8N 基础设施、全局实体字典、碎片暂存目录、审核流程。
区别只在数据入口：一个是自动采集的 AI 对话，一个是手动投喂的各类素材。

## 十一、风险与防护

| 风险 | 防护 |
|------|------|
| 原始数据膨胀（含 think 块和 base64） | raw/ 目录排除 Obsidian 索引 |
| N8N 覆写审核中的文件 | 绝对规则：只新建，不覆写 |
| Git 合并冲突 | 按设备分文件，时间戳命名 |
| Tampermonkey 脚本因 Gemini 改版失效 | 预期会发生，修复成本低（几十行 JS） |
| 审核积压 | 快速扫描策略（30秒/个分流） |
