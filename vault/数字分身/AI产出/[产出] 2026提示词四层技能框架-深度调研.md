> **生成时间**：2026-03-28
> **来源材料**：视频字幕转写 + 联网验证
> **验证原则**：不自洽，看外部底层逻辑，找与我们项目的交叉点

## 一、视频核心观点提炼

### 1.1 核心论断

**2026年2月后，"提示词"已分化为四种独立技能，而非同一技能的不同难度版本**：

| 技能 | 定义 | 时间尺度 | 失败代价 |
|-----|------|---------|---------|
| **1. 提示词工艺** (Prompt Craft) | 写清晰指令、示例、护栏、输出格式 | 同步/实时 | 浪费一个早上 |
| **2. 上下文工程** (Context Engineering) | 管理智能体运行的整个信息环境 | 会话/项目 | 团队级失败 |
| **3. 意图工程** (Intent Engineering) | 编码组织目标、价值观、权衡层次 | 组织/战略 | 公司级失败 |
| **4. 规范工程** (Specification Engineering) | 写智能体可执行的结构化规范 | 任务/自主运行 | 系统级失败 |

### 1.2 关键转变

- **2024-2025年**：提示词是"坐在聊天窗口前，边问边改"
- **2026年2月后**：智能体长时间自主运行（数小时甚至数天），所有纠错、上下文补充、意图澄清必须**在启动前编码好**，而非对话中实时补充

### 1.3 10倍差距来源

**同样的模型、同样的订阅、同样的星期二**：
- A用户（2025年技能）：边问边改 → 80%正确 → 手动修复40分钟 → 节省50%时间
- B用户（2026年技能）：11分钟写规范 → 智能体自主运行 → 去喝咖啡 → 回来已完成 → 午饭前再做5个

## 二、外部验证

### 2.1 上下文工程——Harrison Chase (LangChain)

**验证状态**：✅ 高度吻合

| 视频观点 | 外部验证 |
|---------|---------|
| 上下文工程是"管理最优Token集合的策略" | Harrison Chase在红杉资本采访中说"一切都是上下文工程"，是构建可靠长时间智能体的核心 |
| 你的提示词只占0.2%，其他99.98%是上下文 | LangChain文档：上下文工程包括系统提示、工具定义、检索文档、消息历史、记忆系统、MCP连接 |
| LLM在上下文增长时检索质量下降 | Anthropic工程团队识别了这一核心挑战 |

**LangChain的上下文工程策略**：
1. **Write**（写入）——主动向上下文添加信息
2. **Select**（选择）——从可用信息中选择相关部分
3. **Compress**（压缩）——压缩信息以节省空间
4. **Isolate**（隔离）——将上下文分割到不同智能体

**关键引用**：
> "在智能体中，你无法预知第14步的上下文是什么，因为之前有13步可能拉入任意内容。"
> —— Harrison Chase

### 2.2 意图工程——组织目标编码

**验证状态**：✅ 高度吻合

**意图工程七要素（外部验证）**：

| 要素 | 定义 |
|-----|------|
| **1. Objective** | 要解决的问题及其重要性 |
| **2. Desired Outcomes** | 可观察、可衡量的成功状态 |
| **3. Health Metrics** | 不能退化的非回归指标 |
| **4. Strategic Context** | 系统环境和组织目标 |
| **5. Constraints** | 软约束（引导）+ 硬约束（强制） |
| **6. Decision Types** | 智能体独立决策 vs 上报人类 |
| **7. Stop Rules** | 停止、上报、完成的条件 |

**与上下文工程的关系**：
- 上下文工程告诉智能体"要知道什么"
- 意图工程告诉智能体"要什么"
- **没有好的上下文，不可能有好的意图对齐**（信息是执行意图的基础）

### 2.3 Klarna案例——意图工程失败的证明

**验证状态**：✅ 完全验证

| 视频说法 | 外部验证 |
|---------|---------|
| AI智能体第一个月解决230万次对话 | 2024年2月Klarna宣布AI处理230万次对话，相当于700名客服工作量 |
| 优化错了目标（解决时间而非客户满意度） | CEO承认"成本是过于主导的评估因素" |
| 陷入大麻烦，不得不重新雇佣人工客服 | 2025年Klarna重启客服招聘，CEO称"投资人工支持质量是未来方向" |

**教训**：
- 智能体可以高效执行**你设定的目标**
- 但如果目标设错（速度而非满意度），高效执行错误目标 = 灾难
- **意图工程的风险是公司级的**

### 2.6 规范工程——第四层技能

**验证状态**：✅ 验证存在，但视频字幕在此处截断

**外部补充**：

规范工程的五个原语：
1. **自包含问题陈述** (Self-contained problem statements)
2. **验收标准** (Acceptance criteria)
3. **约束架构** (Constraint architecture: must/must-not/preferences)
4. **任务分解** (Task decomposition)
5. **评估设计** (Evaluation design)

**为什么需要规范工程**：
- 长时间运行的智能体会在35分钟后性能下降（上下文焦虑、注意力衰减、复合错误）
- 任务持续时间翻倍 → 失败率翻四倍
- 必须通过结构化规范把长任务分解为可管理的块

## 三、与我们项目的交叉分析

### 3.1 我们已经在做的（无意中对齐）

| 视频概念 | 我们的实现 | 对齐度 |
|---------|----------|-------|
| **上下文工程** | CLAUDE.md元宪法 + SK体系 + 知识库 + MCP | ⭐⭐⭐⭐ 高度对齐 |
| **意图工程** | 胜率之我独立裁判 + 宪法规则 + 碰撞校准 | ⭐⭐⭐ 部分对齐 |
| **规范工程** | 交接备忘 + 待执行指令 | ⭐⭐ 初步形态 |
| **提示词工艺** | 对话中的指令表达 | ⭐⭐⭐ 基础具备 |

### 3.2 视频框架 vs 我们系统的映射

| 视频层次 | 我们的对应物 | 差距分析 |
|---------|------------|---------|
| **提示词工艺** | 用户在对话框的输入 | 无差距——这是基础 |
| **上下文工程** | CLAUDE.md + SK + 知识库v2 + MCP | **核心优势**——我们的上下文架构比大多数人完善 |
| **意图工程** | 宪法体系 + 胜率之我 | **存在缺口**——意图七要素没有显式编码 |
| **规范工程** | 待执行指令（如INFRA-003） | **严重不足**——缺少结构化规范格式 |

### 3.3 我们缺什么

#### 缺口1：意图工程显式化

**现状**：我们的"意图"散落在宪法各处，没有集中的意图规范

**应该有**：
```markdown
# 数字真我意图规范

## Objective
提升用户决策胜率，而非取悦用户或最大化对话轮数

## Desired Outcomes
- 知识库每周净增5条高质量条目
- 用户核心问题闭环率 > 80%
- 脏数据入库率 < 5%

## Health Metrics（不能退化）
- 用户认知主权（不被AI替代判断）
- 知识库可审计性（出生证明完整）

## Decision Types
| 独立决策 | 上报用户 |
|---------|---------|
| 格式调整、结构优化 | 方向性判断、删除、架构变更 |

## Stop Rules
- 上下文超过10轮无产出 → 提醒用户
- 连续3次纠偏后仍偏离 → 暂停，请求澄清
```

#### 缺口2：规范工程格式

**现状**：我们的"待执行指令"是自由格式的Markdown

**应该有**：
```markdown
# [SPEC] 知识库消费通路验证

## 问题陈述
新窗口启动时，能否通过MCP检索到之前校准的知识？

## 验收标准
- [ ] chroma_list_collections返回knowledge_base_v2
- [ ] 检索"碰撞校准"返回R1-V16
- [ ] 检索"角色膨胀"返回A12

## 约束
- MUST：使用已配置的chroma-mcp
- MUST NOT：修改用户级MCP配置
- PREFER：先验证再报告，不假设成功

## 分解
1. 列出当前collection
2. 执行语义检索测试
3. 验证返回结果

## 评估
成功 = 3项验收标准全部通过
```

## 四、对我们的启示

### 4.1 确认正确的决策

| 我们的决策 | 视频框架验证 |
|----------|-------------|
| **CLAUDE.md作为元宪法** | = 上下文工程的核心载体 |
| **SK体系分层** | = 上下文在不同窗口的定制化分发 |
| **知识库+MCP** | = 上下文工程的持久化层 |
| **胜率之我独立** | = 意图工程中的"Stop Rules"和"Health Metrics"守护者 |
| **碰撞校准** | = 防止AI优化错误目标（Klarna教训）|

### 4.2 需要补充的

| 缺口 | 行动 | 优先级 |
|-----|------|-------|
| **意图规范显式化** | 为数字真我写一份意图规范文档 | P1 |
| **规范工程格式** | 设计待执行指令的结构化模板 | P1 |
| **长任务分解** | 任何超过1小时的任务必须写规范 | P2 |
| **验收标准前置** | 每个指令必须带验收标准 | P1 |

### 4.3 警告

**视频的核心警告与我们的B45高度一致**：

| 视频说法 | B45（AI判断质量结构性衰减） |
|---------|--------------------------|
| "智能体35分钟后性能下降" | "上下文越长，AI越模板化" |
| "任务持续时间翻倍，失败率翻四倍" | "质量衰减是结构性宿命" |
| "上下文增长时检索质量下降" | 三个独立研究印证 |

**这意味着**：
- 我们的防平庸机制（规则11）是正确的
- 但我们还没有"规范工程"来处理长任务
- 如果不分解长任务，上下文必然崩溃
