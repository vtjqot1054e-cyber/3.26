```
果：AI给了什么结果
用户反应：好/坏/中性
结论：下次遇到类似场景应该/不应该这么做
边界：什么条件下成立
```

## 二、对胜率之我的需求

### 审什么

| 对象 | 说明 |
|------|------|
| **知识库条目** | 每条是否有完整三元结构+因果链+出生证明 |
| **定性准确性** | 用户定性（好/坏/中性）是否被正确识别，是否有误判 |
| **边界标注** | 结论的成立条件是否清晰，是否有过度归纳 |

### 怎么审

**建议流程**：
```
读条目
  → 检查三元结构完整性（AI行为/导致/用户反应）
    → 检查因果链（因/果/用户反应/结论/边界）
      → 检查出生证明（用户原话+行号+定性）
        → 给出审校意见（通过/修改/删除）
```

**通过线**：
- 三元结构完整
- 因果链有边界标注
- 出生证明有用户原话+来源文件+行号
- 用户定性已标注（好/坏/中性）

### 审完产出什么

| 产出 | 去向 |
|------|------|
| 审校意见 | 留痕到`vault/knowledge_base/inbox/胜率之我-*.md` |
| 通过/修改/删除建议 | 交给用户终审 |
| 用户终审后 | 数字真我执行入库（唯一写入者） |

**分工明确**：
- 胜率之我 = 审校意见 + 证据链 + 通过线（**不做入库**）
- 数字真我 = 知识库唯一写入
- 用户 = 终审

## 三、当前痛点（旧审计方式的问题）

### 痛点1：审的是结论，不是因果链

旧方式：检查结论是否"正确"
问题：没有因果链的结论是死知识，不知道什么条件下成立

### 痛点2：没有出生证明

旧方式：结论直接入库
问题：追溯不到原始对话，无法验证结论是否被正确提炼

### 痛点3：用户定性缺失

旧方式：AI自己判断好/坏
问题：AI可能误判用户信号（愤怒→教导、批评→共鸣）

### 痛点4：审完不知道谁执行

旧方式：胜率审完后不清楚谁来改
问题：V15-B4已明确分工，但胜率SK v1没有体现这个分工
