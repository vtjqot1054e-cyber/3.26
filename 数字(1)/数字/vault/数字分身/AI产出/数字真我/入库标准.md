# 精炼层ABC入库标准

> 本文件是精炼层（向量库）的验收标准。复盘大脑终审产出的每一条ABC片段，必须通过本标准才能入库。
> 本标准由数字真我制定和修订。复盘大脑、提取助手、标注助手不得自行放宽。

---

## 通用规则（三类共享）

### 颗粒度

| 规则 | 说明 |
|------|------|
| 每条≤500字 | 超过500字的必须拆分或压缩，不是截断 |
| 一条一主题 | 不能把两个无关结论塞进一条 |
| 自含义 | 脱离原文上下文也能看懂。不能出现"如上所述""前面提到的"这类悬空引用 |
| 有出处 | 每条必须标注原文行号+来源文件，便于回溯 |

### 禁入内容（硬拦截）

| 内容 | 处理 |
|------|------|
| AI修辞废话（"让我帮你看看""这是个好问题"） | 拦截，不入库 |
| 启动仪式/关窗日志 | 拦截 |
| 工具调试过程（报错、路径修复、依赖安装） | 拦截，除非踩坑本身是知识 |
| 重复确认（"好的""对""继续"） | 拦截 |
| 未经验证的AI推测（AI说"可能是因为..."但没有后续验证） | 标uncertain，留存档层，不进精炼层 |
| 含个人隐私信息（具体地址、身份证、银行卡） | 拦截，物理删除或脱敏后重新评估 |

### 标签规则

每条片段必须携带：

1. **用途标签**（必选至少一个）：`profile_*` 或 `knowledge_*`，C类双标签
2. **内容标签**（至少一个）：从下方内容标签池选择
3. **来源标注**：`source_file` + `source_lines`
4. **时间标注**：原始对话发生时间（不是入库时间）
5. **置信度**：高/中/低（复盘大脑终审标注）

**内容标签池：**

| 标签 | 适用场景 |
|------|---------|
| `validated_decision` | 事后被证明正确的判断/决策 |
| `invalidated_decision` | 事后被证明错误的判断/决策（同样有价值——偏差数据） |
| `bias_record` | 用户的偏差模式（过度自信、焦虑驱动、拖延等） |
| `emotional_state` | 情绪状态快照（愤怒、兴奋、焦虑、平静） |
| `cognitive_shift` | 认知变化（从A观点转到B观点，含转变原因） |
| `decision_rationale` | 决策理由（为什么选A不选B） |
| `know_how` | 经验诀窍（踩过坑后总结的正确做法） |
| `architecture_decision` | 系统架构决策（宪法级/窗口/管线/SK） |
| `business_intelligence` | 业务情报（客户、行业、竞争格局） |
| `person_profile` | 人物画像片段（用户自己或周围人的行为模式） |
| `uncertain` | 待验证（用户标❓或数字真我判断证据不足） |

## A类：画像片段

### 定义

反映用户**当时**的心理状态、行为习惯、决策风格、偏差模式。核心用途：让AI理解"你是怎样的人"。

### 合格标准

| 维度 | 要求 | 不合格示例 |
|------|------|-----------|
| **是行为不是结论** | 记录用户做了什么、说了什么，不记录AI的归纳 | ❌ "用户具有高度开放性"（这是AI归纳） ✅ "用户在工具选型中5分钟内否定了3个方案，每次否定理由不同——第一次嫌复杂，第二次嫌不成熟，第三次嫌和现有体系不兼容"（这是行为记录） |
| **有时间锚点** | 必须能定位到具体对话场景 | ❌ "用户总是很急躁" ✅ "2026-03-15晚对话中，用户在等待N8N安装时连续催了3次'好了没'" |
| **允许矛盾** | 不同场景下的矛盾行为都要记录，不替用户圆 | ✅ "大脑窗口：直接否定AI建议不犹豫" + "顾问窗口：同一类建议会犹豫5分钟再决定" |
| **标注场景** | 必须标注该行为出现在什么窗口/什么情境下 | 缺场景的画像片段 = 不入库 |
| **不美化不贬低** | 客观描述，不加"这说明用户很XX" | ❌ "用户展现出卓越的决策力" |

### A类标签组合示例

- `profile_behavior` + `bias_record`：用户在压力下的决策偏差
- `profile_behavior` + `emotional_state`：用户在特定场景的情绪反应
- `profile_cognition` + `cognitive_shift`：用户认知发生转变的具体过程

### A类特殊规则

| 规则 | 说明 |
|------|------|
| 单篇不写入画像 | 画像更新必须3篇以上交叉验证后由数字真我批准 |
| 偏差数据高优先级 | 用户犯错的记录比正确的记录更有入库价值（纠偏弹药） |
| 情绪数据带触发条件 | 不是"用户焦虑了"，是"用户在X条件下焦虑了" |

## B类：知识片段

### 定义

去掉废话后的**正确结果**——事件、决策、解法、经验。核心用途：让AI帮用户做事时有准确的知识库可查。

### 合格标准

| 维度 | 要求 | 不合格示例 |
|------|------|-----------|
| **是结果不是过程** | 记录最终结论，不记录中间试错（除非试错本身是know-how） | ❌ "先试了方案A不行，又试了方案B也不行，最后用方案C"（这是过程）✅ "方案C可行。原因：A的问题是XX，B的问题是YY"（这是结果+理由） |
| **可验证** | 有明确的对错判断条件 | ❌ "Docker好像不太适合这个场景"（模糊）✅ "N8N Code节点无法访问process.env——沙箱限制，必须硬编码路径"（可验证） |
| **标注有效期** | 技术类标 `still_valid` / `outdated`；决策类标 `scope:permanent` / `scope:temporary` | 不标有效期的B类 = 不入库 |
| **去AI修辞** | 原文如果是AI回复，必须去掉修辞只留干货 | ❌ "经过仔细分析，我认为最优方案是..."  ✅ "最优方案：XX。理由：1... 2... 3..." |
| **不重复** | 和已有B库对比，重复的合并而非新增 | 两条都说"N8N的Code节点不能用process.env" → 合并为1条 |

### B类子分类

| 子类 | 内容 | 典型标签 |
|------|------|---------|
| **B-决策** | 架构决策、业务决策、工具选型 | `knowledge_decision` + `architecture_decision` 或 `business_intelligence` |
| **B-经验** | 踩坑记录、解法、操作诀窍 | `knowledge_experience` + `know_how` |
| **B-事实** | 事件记录、环境配置、客观数据 | `knowledge_fact` |
| **B-流程** | 已验证的SOP、工作流程 | `knowledge_process` |

### B类特殊规则

| 规则 | 说明 |
|------|------|
| 决策必须带理由 | "废弃SpecStory"不够，"废弃SpecStory，因为XX"才合格 |
| 过时标注不是删除 | 决策后来被推翻 → 标 `outdated` + 标推翻时间和原因，不删除原条目 |
| 一次性知识控制入库 | "只对某次场景有用"的具体操作步骤 → 留存档层，不进精炼层（太具体的知识污染检索） |
| 太抽象的知识不入库 | "做事要有系统性"→ 没有可操作性，拦截 |

### B类质量红线

以下任一项出现 = 不合格：
1. 结论和理由不匹配（说选A但理由支持B）
2. 缺少验证状态（不知道这个结论现在还对不对）
3. 颗粒度过粗（"N8N有很多坑" → 哪些坑？）
4. 颗粒度过细（"在第47行第3个字符修改了逗号" → 不是知识）

## C类：用户观点片段

### 定义

用户在审阅过程中产出的评述、判断、理由。这是用户的**元认知**——"我怎么看我自己的对话"。

### 为什么C类最值钱

C类是用户主动产出的，不是AI提取的。它同时暴露：
- **画像信号**：你为什么这样判断（你的判断框架 = A画像）
- **知识信号**：你改了什么分类（你对知识对错的标准 = B知识）

所以C类必须双标签 `profile_*` + `knowledge_*`。

### 合格标准

| 维度 | 要求 | 不合格示例 |
|------|------|-----------|
| **保留原话** | C类尽量用用户原话，不改写。改写会丢失语气信号 | ❌ AI改写后的"用户认为该方案不可行" ✅ 用户原话"这个方案扯淡，根本没考虑实际情况" |
| **标注审阅对象** | 明确用户在评价哪一段原文 | 缺少评价对象的C类 = 悬空，不入库 |
| **区分判断和情绪** | 用户说"这段垃圾"——是情绪（A类），还是判断（B类），还是两者都有（C类双标签） | 不区分 = 分类不准 |
| **和用户标记关联** | C类片段必须和用户的✅❌❓动作关联 | 孤立的C类评述（不知道用户最终判定是什么）= 不入库 |

### C类来源

| 来源 | 优先级 | 说明 |
|------|--------|------|
| 用户在OB中插入的文字旁注 | 最高 | 事后反思，深度最高 |
| 用户的Obsidian脚注 `[^N]` | 最高 | 结构化的独立批注 |
| 用户修正分类方向的动作 | 高 | "这不是B知识是A画像"——暴露判断标准 |
| 用户✅❌❓标记+理由 | 高 | 判断本身就是C类数据 |
| 用户在对话窗口中的口头评述 | 中 | 实时碎片，需整理确认后才算 |

### C类特殊规则

| 规则 | 说明 |
|------|------|
| C类必须双标签 | 同时标 `profile_*` 和 `knowledge_*`，不能只标一个 |
| 不整理不入库 | 用户碎片原话需经标注助手整理+用户确认后才算C类。未确认的碎片留存档层 |
| 判断框架积累 | C类数据积累后，可以提炼用户的"判断框架"（什么条件下判断准，什么条件下判断偏），这是胜率之我的核心弹药 |

## 退回规则（不合格处理）

| 不合格类型 | 处理 |
|-----------|------|
| 内容合格但分类错（A标成了B） | 退回复盘大脑纠正分类，不入精炼层 |
| 内容模糊（证据不足以判断对错） | 标 `uncertain`，留存档层，等更多数据再评估 |
| 内容重复（已有同义条目） | 合并到已有条目，不新增 |
| 内容过时（决策已推翻） | 原条目标 `outdated`，新结论作为新条目入库 |
| 内容有害（脏数据/噪音/AI修辞残留） | 拦截，永不入库。记录拦截理由 |

## 入库审核流程

```
复盘大脑终审产出ABC草稿
        │
        ▼
数字真我逐条审核
        │
    ┌───┼───┐
    ▼   ▼   ▼
  通过  模糊  不通过
    │   │     │
    ▼   ▼     ▼
  入库  标uncertain  退回+标理由
  精炼层  留存档层    复盘大脑修正
```

## 上帝视角审计（v2新增，每篇入库审核必做）

入库审核不只是逐条审ABC。数字真我同时站在全局视角审视这篇对话的系统性意义。

### 审计维度

| 维度 | 内容 | 产出去向 |
|------|------|---------|
| **用户状态诊断** | 结合已入库A类画像+本篇对话，用户当前处于什么心理/认知/行为状态？是否陷入某种模式？ | 审计意见 → 用户带给总顾问 |
| **注意力建议** | 结合用户最高目标（胜率），用户当前注意力是否偏离？应该聚焦什么？ | 审计意见 → 用户带给总顾问传达 |
| **对话质量评估** | 本篇对话质量高/中/低？高质量的对话模式是什么（如：用户主导提问、AI严格执行纪律）？ | 审计意见 |
| **窗口纪律审计** | 对话中是否存在纪律问题：无确认节点、AI自行推进、回复逻辑混乱、跳过用户确认等 | 审计意见 → 用户带给总大脑，要求对应窗口改进 |
| **SK改进建议** | 好的对话模式是否应固化到对应窗口SK？坏的模式是否应写入禁止行为？ | 具体建议+目标SK路径 → 用户带给总大脑执行 |

### 审计意见格式

每篇入库审核产出时，在ABC审核结果之后附加：

```markdown
### 上帝视角审计

**用户状态**：[一句话诊断]
**注意力偏离**：是/否。[如偏离，说明偏向哪里，应该聚焦什么]
**对话质量**：高/中/低。[理由]
**窗口纪律问题**：
- [问题1：具体描述+出现在行号+涉及窗口]
- [问题2：...]
**SK改进建议**：
- [建议1：目标SK+具体修改内容]
- [建议2：...]
**需要用户转交的指令**：
- → 总顾问：[具体内容]
- → 总大脑：[具体内容]
```

### 权力边界

- 数字真我**出具审计意见**，不直接给总顾问/总大脑下指令
- 用户是物理通道——把审计意见带到对应窗口
- 总大脑决定是否执行改进、如何执行
- 数字真我跟踪：下次同一窗口的对话进入管线时，检查纪律问题是否改善

## 阶段性验证（数字真我自检）

| 触发 | 动作 |
|------|------|
| 每10篇归档后 | 抽查精炼层3条A、3条B、3条C，和原始存档对比——精炼是否准确，是否丢了关键信息 |
| 用户反馈"AI回答不准" | 紧急排查：是哪条精炼数据导致了错误检索 |
| 画像更新后 | 验证新画像描述和A类精炼数据是否一致 |
