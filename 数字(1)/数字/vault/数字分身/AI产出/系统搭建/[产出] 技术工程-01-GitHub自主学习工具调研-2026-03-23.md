# GitHub AI Agent 自主学习/技能提取工具调研

> 技术工程-01 执行，2026-03-23。用户指令：调研 Claudeception 及类似工具，评估能否应用到数字分身项目。

---

## 一、调研背景

用户发现 Claudeception 工具后，要求系统搜索 GitHub 上所有类似的 AI Agent 自主学习/知识提取工具，评估哪些理念和机制可以借鉴到数字分身项目的知识沉淀流程中。

## 三、逐项详细分析

### 3.1 Acontext（memodb-io）

**工作方式**：
```
会话消息 → 任务完成/失败 → LLM 蒸馏 → Skill Agent 路由 → 更新 Markdown 文件
```

**召回方式**：不用向量搜索，Agent 通过 `get_skill` / `get_skill_file` 工具调用主动获取——"progressive disclosure, agent in the loop"。

**架构**：需要后端（PostgreSQL + Redis + RabbitMQ + S3），支持自托管（Docker）。

**可借鉴**：
- "Skill = Memory" 理念：知识不是被动存储等搜索，而是 Agent 主动推理需要什么再去取
- 蒸馏环节的设计：从会话中提取"什么有效、什么失败、用户偏好"三个维度
- 支持用户自定义 schema——用 SKILL.md 定义记忆的结构

**不适用**：后端架构太重，我们不需要这个量级的基础设施。

---

### 3.2 agent-memory-skill（smixs）⭐ 与我们契合度最高

**核心**：模拟人类遗忘曲线（Ebbinghaus 1885），一个 Python 文件，零依赖，直接操作 Markdown 文件的 YAML frontmatter。

**三层架构**：

| 层 | 内容 | 对应我们的 |
|---|---|---|
| Hot Context（<4KB） | 当前焦点、阻塞项、提醒 | 交接备忘 |
| Searchable Vault | 带 frontmatter 的卡片，分 5 级 | 知识库层1（精炼 MD） |
| Archive | 旧日志、完成的项目 | 知识库层3（有价值全文） |

**遗忘曲线机制**：
```
relevance = max(0.1, 1.0 - days × 0.015)
```

| 层级 | 距上次访问天数 | relevance | 行为 |
|------|-------------|-----------|------|
| core | 手动设定 | 1.0 | 永不自动降级（身份、安全、定价） |
| active | 0-7 天 | 1.0-0.90 | 所有模式可搜 |
| warm | 8-21 天 | 0.89-0.69 | 默认搜索 |
| cold | 22-60 天 | 0.68-0.10 | 仅深度搜索 |
| archive | 60+ 天 | 0.10（底线） | 仅创意模式或显式召回 |

**Graduated Touch（间隔重复）**：
```
archive → cold → warm → active → active（刷新）
```
每次访问只升一级，多次访问 = 更强记忆。

**Creative Mode**：随机从 cold/archive 拉卡片，模拟"灵光一闪"——用户可能在旧知识中发现新关联。

**搜索模式**：

| 模式 | 搜索范围 | 场景 | Token 消耗 |
|------|---------|------|-----------|
| heartbeat | core + active | 快速状态检查 | ~2K |
| normal | active + warm | 日常任务 | ~5K |
| deep | 全部 | 战略分析 | ~15K |
| creative | 随机 cold+archive | 头脑风暴 | ~3K |

**为什么与我们高度契合**：
1. 三层架构几乎一一对应我们的知识库三层
2. 纯 Markdown + YAML frontmatter，和 Obsidian 完美兼容
3. 零依赖，不需要额外服务，直接在现有文件上运行
4. 遗忘曲线解决了我们未来"知识库越来越大、什么都同等权重"的问题
5. core 层级 = 我们的核心画像（永不降级）
6. Creative Mode 可以辅助复盘大脑做交叉验证时发现意外关联

### 3.3 Homunculus v2（humanplane）

**核心创新**：不直接提取 Skill，而是先学习"本能"（instinct）——原子级行为规则。

**本能模型**：
```yaml
trigger: "when writing new functions"
confidence: 0.7
domain: "code-style"
action: "Use functional patterns over classes."
evidence: "Observed 5 instances of functional pattern preference."
```

**进化路径**：
```
Hook 观察（100%可靠）→ 后台 Agent 分析 → 生成本能（自动）→ 本能聚类 → 用户手动 /evolve → 生成 Skill/Command/Agent
```

**v1 vs v2 关键改进**：

| | v1 | v2 |
|---|---|---|
| 观察 | Skill 触发（50-80% 概率） | Hook 触发（100%） |
| 分析 | 主上下文 | 后台 Agent（Haiku，廉价模型） |
| 粒度 | 直接生成大 Skill | 先生成原子本能，再聚合 |
| 置信度 | 无 | 0.3-0.9，被反驳则衰减 |

**可借鉴**：
- "本能 → 聚类 → Skill" 的渐进式学习比直接提取更稳健——类似我们的"积累够了再写入核心画像"
- confidence 衰减机制（被反驳则降低置信度）——和我们的交叉验证逻辑一致
- domain 标签分类——和我们的 ABC 分类思路一致
- 后台用廉价模型分析，不占主上下文——我们可以考虑类似思路

**不适用**：Claude Code 专属插件架构，不能直接用于 Cursor。

### 3.4 claude-code-semantic-memory（zacdcook）

**工作方式**：
```
.jsonl 会话记录 → Markdown → 子 Agent 提取 learnings → 本地嵌入（nomic-embed-text）→ 每次 prompt 注入相关记忆
```

**学习类型分类**：

| 类型 | 说明 | 对应我们的 |
|---|---|---|
| WORKING_SOLUTION | 确认有效的命令/模式 | B 知识 |
| GOTCHA | 陷阱和反直觉行为 | B 知识 |
| PATTERN | 反复出现的架构决策 | B 知识 |
| DECISION | 明确的设计选择+理由 | B/C 知识/观点 |
| FAILURE | 什么没用+为什么 | B 知识（教训） |
| PREFERENCE | 用户偏好 | A 画像 |

**关键设计决策**：
- "Hook 优于 CLAUDE.md"——Hook 确定性触发，CLAUDE.md 指令在认知负载高时可能被跳过
- nomic-embed-text 优于 MiniLM——8K token 上下文 vs 256 token，长对话不会被截断
- 0.45 相似度阈值——宽松到能捕获语义相关内容，不至于噪音太多

**可借鉴**：
- 学习类型分类和我们的 ABC 有对应关系，PREFERENCE 类型提醒我们画像信号也可以自动提取
- 上下文压缩时自动导出+提取的机制——类似我们的"关窗前总结"但更自动化

**不适用**：需要 Ollama 本地嵌入服务，Claude Code 专属 Hook 系统。

---

### 3.5 Claudeception（blader）

**工作方式**：Hook 注入提醒 → Claude 完成任务后自评"是否有可提取知识" → 满足质量门控则生成 Skill 文件。

**质量门控四条标准**：
1. 非显而易见（需要真正的调查发现，不是查文档）
2. 可复用（未来任务也能用）
3. 有具体触发条件（能被语义匹配到）
4. 已验证有效（不是理论上的）

**可借鉴**：
- 质量门控标准简洁有效，可以作为我们 B 类知识入库的快速筛选参考
- description 字段的写法指导——"Helps with database problems" 太模糊，"Fix for PrismaClientKnownRequestError in serverless" 才能被匹配到——对我们 SK 的 description 写法有参考价值

---

### 3.6 continuous-learning（affaan-m）

**工作方式**：Stop Hook 在会话结束时评估，提取 5 类模式。

**5 类模式**：error_resolution / user_corrections / workarounds / debugging_techniques / project_specific

**特点**：轻量，但粒度粗（只在会话结束跑一次）。

**可借鉴**：`user_corrections` 这个类型值得注意——用户纠正 AI 的行为本身就是高价值信号，我们的管线已经在做这件事（用户审阅时的删除线/高亮就是 correction）。

## 四、与我们现有体系的对比总结

| 维度 | 我们现有 | 这些工具的共同理念 | 差距/可改进 |
|------|---------|-------------------|-----------|
| **知识提取** | 人工驱动（用户审阅→复盘大脑→数字真我） | 自动/半自动提取 | 我们的质量更高但效率低 |
| **知识存储** | MD + JSON + 向量库三层 | 大多用 Markdown，少数用向量库 | 我们的架构更完整 |
| **知识召回** | 各窗口手动读知识库 | Hook 注入 / Agent 工具调用 / 语义匹配 | **我们缺少自动召回机制** |
| **时间衰减** | 无（所有知识同等权重） | agent-memory-skill 有遗忘曲线 | **我们缺少时间维度** |
| **质量控制** | 三层审核（用户→复盘→数字真我） | 单层自评或无 | 我们远优于它们 |
| **跨会话持久** | 交接备忘 + 知识库文件 | Hook + 持久化存储 | 我们已有，但不够自动化 |

## 五、可借鉴机制清单（建议大脑评估优先级）

### 5.1 遗忘曲线 + 分级机制（来源：agent-memory-skill）

**机制**：给知识库每条精炼加 `relevance` 分数和 `tier` 层级，随时间自动衰减，被访问时升级。

**适用场景**：知识库从 58 条增长到 500+ 条时，避免"什么都同等权重"导致检索噪音。

**实现难度**：低。只需在现有 MD 文件的 YAML frontmatter 加 `last_accessed` / `relevance` / `tier` 字段，写一个 Python 脚本定期跑 decay。

**风险**：当前 58 条还不急，但架构上预留是值得的。

### 5.2 工程窗口"任务后自评"环节（来源：Claudeception + Homunculus）

**机制**：工程窗口完成任务后，强制自评"本次是否有可提取的技术知识"。如果有，按模板写入 `knowledge_base/inbox/`。

**适用场景**：交接备忘里的"教训"条目（如 `process.env 沙箱不可用`、`N8N 空数组不触发下游`）目前是手动记录，经常遗漏。

**实现难度**：低。在工程操作纪律 SK 中加一个"关窗前自评"环节即可，不需要 Hook。

**风险**：AI 自评有偏差（数字真我-11 已指出），需要数字真我后续审核。

### 5.3 SK description 语义优化（来源：Claudeception）

**机制**：SK 的 description 字段要写得足够具体，包含触发条件、错误信息、场景关键词，才能被准确匹配。

**适用场景**：我们现有 SK 的 description 有些过于笼统。

**实现难度**：低。技术总工审计时顺手优化。

### 5.4 本能式渐进学习（来源：Homunculus v2）

**机制**：不直接写入核心画像，而是先记录"本能"（原子级观察+置信度），积累到阈值后再聚合。

**适用场景**：我们的 A 类画像更新已经在用类似逻辑（"积累够了再写入核心画像"、"交叉验证达标才入库"），但没有形式化。

**实现难度**：中。需要设计"画像信号"的标准格式和置信度评分体系。

**风险**：过度形式化可能增加复杂度，当前管线冻结中不宜大改。

### 5.5 知识自动召回/注入（来源：claude-code-semantic-memory + Acontext）

**机制**：窗口启动时或任务执行中，自动从知识库检索相关知识注入上下文。

**适用场景**：目前各窗口需要手动读知识库，容易遗漏。

**实现难度**：高。Cursor 没有 Claude Code 的 Hook 系统，需要其他方式实现（如在 SK 的启动流程中加入知识库检索步骤）。

**风险**：自动注入可能引入噪音，需要精确的相关性判断。

## 六、工程建议（供大脑决策）

### 短期可做（不改架构，不违反管线冻结令）

1. **在工程操作纪律 SK 中加"任务后自评"环节**——模板化，写入 inbox
2. **技术总工下次审计时优化各 SK 的 description 字段**

### 中期考虑（管线冻结解除后）

3. **引入 agent-memory-skill 的遗忘曲线机制**——给精炼 MD 加 frontmatter 字段，写 decay 脚本
4. **在各窗口 SK 的启动流程中加"知识库相关条目检索"步骤**

### 长期方向（需要更多数据验证）

5. **形式化"画像信号"的置信度评分体系**——借鉴 Homunculus 的 instinct 模型
6. **探索 Cursor 环境下的自动知识注入方案**
