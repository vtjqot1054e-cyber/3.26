> 上级：总大脑

---

## 内部优先检查

| 工具 | 状态 | 说明 |
|------|------|------|
| SpecStory | 大脑-07废弃"全面删除不再提及" | 但本次是总大脑指派评估其已有产出（52个history文件）的可利用性，不是评估是否重新启用 |

## 一、实际查证：两种格式对比

### Export Transcript格式（手动右键导出）

```markdown
# 总顾问（大脑常驻层）4.6MAX
_Exported on 2026/3/17 at GMT+8 20:09:01 from Cursor (2.6.19)_

---

**User**

启动本项目，开启常驻层，与仓库同步

---

**Cursor**

我来执行启动流程。先启动工具，然后读取所有必读文件。
```

**特征**：
- 头部：`# 标题` + `_Exported on 时间 from Cursor (版本)_`
- 角色标记：`**User**` / `**Cursor**`
- 分隔：`---`
- ✅ **不含**thinking块
- ✅ **不含**系统注入XML（`<user_query>`、`<open_and_recently>`等）
- ✅ **不含**图片链接/工具调用细节
- 内容是纯对话文本

### SpecStory格式（自动捕获）

```markdown
<!-- Generated by SpecStory, Markdown v2.1.0 -->
<!-- cursor Session UUID (时间) -->

# 标题 (时间)

_**User (时间戳)**_

启动本项目，开启常驻层，与仓库同步

---

_**Agent (model claude-4.6-opus-high, mode Agent)**_

我来执行启动流程。
```

**特征**：
- 头部：HTML注释（SpecStory版本+Session UUID）+ `# 标题 (时间)`
- 角色标记：`_**User (时间戳)**_` / `_**Agent (model XX, mode XX)**_`
- 分隔：`---`
- ✅ **含模型信息**（`model claude-4.6-opus-high`）——Export Transcript没有，这是额外价值
- ✅ **含每条消息的时间戳**——Export Transcript没有
- ❌ **含thinking块**（28/52个文件有）：`<think><details><summary>Thought Process</summary>...内容...</details></think>`
- ❌ **含系统注入XML**（4/52个文件有，早期文件）
- 🟡 图片引用以`[Image]`文本形式出现，不含实际图片数据

## 二、具体差异清单

| 项 | Export Transcript | SpecStory | 差异程度 |
|----|-------------------|-----------|---------|
| **🔴 tool-use块** | **自动剥离，AI文字段合并为连续段落** | **tool-use块与AI文字交替排列，一条AI回复被拆成多段** | 🔴 **最大差异**——49/52文件，1800+次。不是简单删除，需要上下文感知合并 |
| **🔴 AI回复合并** | 同一AI回复的多段文字合并成一段 | 被tool-use分隔成多个独立段落 | 🔴 **高**——过滤脚本的核心难点，需判断两个AI文字段之间只隔tool-use→合并 |
| **thinking块** | 无 | 有（28/52文件，680+次） | 高——需整块删除 |
| **角色标记** | `**User**` / `**Cursor**` | `_**User (时间)**_` / `_**Agent (model, mode)**_` | 中——需正则替换 |
| **头部** | `_Exported on...from Cursor_` | HTML注释 + `# 标题 (时间)` | 中——需替换 |
| **tool-use后的空分隔线** | 无 | tool-use删除后留下连续`---`空行 | 中——需合并 |
| **模型信息** | 无 | 有（`model claude-4.6-opus-high`） | ✅ SpecStory更好 |
| **消息时间戳** | 无 | 有（每条消息） | ✅ SpecStory更好 |
| **系统XML注入** | 无 | 有（4/52文件） | 低——只影响早期文件 |
| **分隔符** | `---` | `---` | ✅ 一致 |
| **正文内容（过滤合并后）** | 纯对话 | 纯对话 | ✅ 一致 |

## 三、过滤脚本需要处理什么

### 按复杂度排序（非简单正则的排在前面）

| # | 过滤项 | 规则 | 出现频率 | 复杂度 |
|---|--------|------|---------|--------|
| 1 | **🔴 tool-use块删除 + AI回复合并** | 删除所有tool-use块；判断两个AI文字段之间如果只隔tool-use（无User消息），则合并为同一段落。**需上下文感知，不是逐行正则** | 49/52文件，1800+次 | 🔴 高 |
| 2 | **thinking块** | 整块删除：从`<think>`到`</think>` | 28/52文件，680+次 | 中 |
| 3 | **Agent角色行改写** | `_**Agent (model XX, mode XX)**_` → `**Cursor**` | 每条AI回复 | 低（正则） |
| 4 | **User角色行改写** | `_**User (时间)**_` → `**User**`（部分无时间戳，需兼容两种格式） | 每条用户消息 | 低（正则） |
| 5 | **SpecStory头部HTML注释** | 删除`<!-- Generated by SpecStory...-->`和`<!-- cursor Session...-->` | 52/52文件 | 低 |
| 6 | **标题行改写** | `# 标题 (时间)` → `# 标题` + 统一头部 | 1次/文件 | 低 |
| 7 | **系统XML标签** | 删除`<user_query>`、`<open_and_recently_viewed_files>`等及其闭合标签和内容 | 4/52文件 | 中（多行匹配） |
| 8 | **tool-use删除后的空分隔线** | 连续多个`---`合并为一个 | 跟随tool-use | 低 |
| 9 | **thinking删除后的空分隔线** | 同上 | 跟随thinking | 低 |

### 过滤脚本核心难点

**#1 tool-use块删除+AI回复合并**是整个脚本的核心难点。Export Transcript做了一件隐藏的事：把被tool-use分隔的同一个AI回复合并成连续段落。

示例（SpecStory原文）：
```
_**Agent (model...)**_
我检测到启动信号，先执行启动工具。
---
<tool-use>... 9个tool调用 ...</tool-use>
---
现在读取核心画像和SOP对话流程。
---
<tool-use>... 6个tool调用 ...</tool-use>
---
全部文件读取完毕。现在我来做接续总结。
```

Export Transcript等效输出：
```
**Cursor**
我检测到启动信号，先执行启动工具。
现在读取核心画像和SOP对话流程。
全部文件读取完毕。现在我来做接续总结。
```

**脚本逻辑**：不能简单删tool-use——还需要检测"两段AI文字之间如果没有User消息，只隔了tool-use/thinking/分隔线"，就把它们合并为同一段（角色行只保留第一个）。

## 四、过滤后能否和Export Transcript一模一样？

**修正结论：过滤+合并后，正文内容可以完全一致。但过滤脚本比简单正则复杂——核心难点是AI回复合并逻辑。**

| 维度 | 能否一致 | 说明 |
|------|---------|------|
| 正文内容 | ✅ **过滤+合并后**完全一致 | 需要tool-use删除+AI回复合并才能达到 |
| 角色标记 | ✅ 过滤后一致 | `**User**` / `**Cursor**` |
| 分隔符 | ✅ 一致 | 都是`---` |
| thinking块 | ✅ 过滤后消除 | 正则删除 |
| tool-use块 | ✅ 过滤后消除 | **但需要上下文感知合并**，不是简单删除 |
| 头部格式 | 🟡 可调整为一致 | SpecStory有模型+时间戳信息可选保留 |

**脚本复杂度评估**：
- 简单正则能搞定的：thinking块删除、HTML注释删除、角色行改写、系统XML删除 → 占过滤项的7/9
- **需要上下文感知的**：tool-use删除+AI回复合并 → 占1/9但是核心难点，需要状态机或分段解析逻辑

**不影响管线的原因不变**：WF1读取frontmatter status+正文。标注助手和信号助手扫描`**User**`/`**Cursor**`交替文本。过滤合并后格式一致，管线不受影响。

## 八、SpecStory数据源机制（2026-03-18 补充）

> 补充来源：总大脑查证 + `.specstory/.what-is-this.md` 第59行

### 数据流程

```
Cursor对话
    ↓ Cursor内部机制
SQLite数据库刷盘（state.vscdb）
    ↓ SpecStory检测到SQLite变化
SpecStory读取SQLite数据
    ↓
生成/更新 .specstory/history/*.md
```

### 关键事实

| 项 | 说明 | 查证来源 |
|----|------|---------|
| 数据源 | Cursor本地SQLite数据库，不是内存或临时文件 | `.specstory/.what-is-this.md` 第59行："Auto-save only works when Cursor or VSCode flushes sqlite database data to disk" |
| 刷盘延迟 | Cursor内部决定何时flush到磁盘，有小延迟 | 同上："This results in a small delay after the AI response is complete" |
| 兜底机制 | 每2分钟强制检查一次 | 同上第69行："or every 2 minutes as a safety net" |
| SpecStory自身 | 只做"读取+格式化输出"，不添加、不修改、不解释对话内容 | 查证：SpecStory MD输出的正文内容和Export Transcript逐字一致 |

### 时间可信度结论

**可信。**

| 时间类型 | 来源 | 可信度 | 理由 |
|---------|------|--------|------|
| Session时间（文件名 `YYYY-MM-DD_HH-MMZ`） | SQLite原始数据 | ✅ 高 | Cursor创建Session时写入数据库，SpecStory原样读取 |
| Session UUID（`<!-- cursor Session UUID -->`） | SQLite原始数据 | ✅ 高 | Cursor生成的唯一标识，SpecStory原样读取 |
| User行时间戳（`_**User (2026-03-11 14:50Z)**_`） | SQLite原始数据 | 🟡 中 | 查证发现：**早期文件（3月11日）所有消息时间戳完全一样**，实际是Session创建时间而非消息发送时间；3月14日及之后大部分文件的User行**没有时间戳** |
| Agent行模型标记（`model claude-4.6-opus-high`） | SQLite原始数据 | ✅ 高 | 每条AI回复独立标记，同一对话可出现不同模型（查证实例：`在哪里弄模型.md` 同时出现 `model default` 和 `model claude-4.6-sonnet-medium-thinking`） |

### 两地git同步对时间的影响

**无影响。**

| 因素 | 分析 |
|------|------|
| 文件系统时间 | git clone/pull会改变文件的mtime/ctime，但SpecStory的时间**写在文件内容里**（文件名+HTML注释+角色行），不依赖文件系统时间 |
| 文件内容 | git同步的是文件内容本身，二进制级别一致，时间数据不会被修改 |
| 结论 | 公司产生的SpecStory文件 → git push → 家里git pull → 内容完全一致，时间信息完整保留 |

## 九、frontmatter元数据保留方案设计

### 当前脚本对元数据的处理（查证）

| 元数据 | 当前脚本处理 | 问题 |
|--------|------------|------|
| Session UUID | **删除**（`RE_SESSION_COMMENT.sub('', text)`，第109行） | ❌ 丢失 |
| Session创建时间 | **删除**（标题行 `(2026-03-17 04:00Z)` 被去掉，第121行） | ❌ 丢失 |
| 模型信息 | **保留在角色行中**（`**Cursor (model xxx)**`） | ✅ 但分散在正文 |
| User行时间戳 | **删除**（`RE_USER_ROLE.sub('**User**', text)`，第176行） | ❌ 丢失 |

### frontmatter格式设计

```yaml
---
source: SpecStory/Cleaned
session_id: 5c0f0dac-8796-429f-920c-adde04bc7906
created: 2026-03-17T04:00:00Z
models:
  - claude-4.6-opus-max
first_message: 2026-03-17T04:00:00Z
last_message: 2026-03-17T12:30:00Z
title: "总顾问（大脑常驻层）4.6MAX"
---
```

#### 字段说明

| 字段 | 来源 | 提取方式 | 备注 |
|------|------|---------|------|
| `source` | 固定值 | — | 标记来源为SpecStory清洗产出 |
| `session_id` | `<!-- cursor Session UUID (时间) -->` | 正则提取UUID | Cursor的Session唯一标识，可用于关联agent-transcripts中的jsonl |
| `created` | 文件名 `YYYY-MM-DD_HH-MMZ` 或 Session注释中的时间 | 正则提取 | Session创建时间（UTC） |
| `models` | 所有Agent角色行中的model参数 | 遍历正文，去重 | 数组形式，一个对话可能用了多个模型 |
| `first_message` | 第一个带时间戳的User行，或回退到`created` | 正则提取 | 🟡 早期文件所有消息时间戳相同=Session时间；后期文件无时间戳=回退到created |
| `last_message` | 最后一个带时间戳的User行，或回退到文件mtime | 正则提取/fallback | 🟡 同上限制。如果无时间戳，可用SpecStory文件的最后修改时间近似 |
| `title` | 标题行 `# 标题 (时间)` | 正则提取去掉时间后缀 | 对话标题/窗口名 |

#### 🔴 不确定项

| 项 | 问题 | 建议 |
|----|------|------|
| `first_message` / `last_message` | 大部分文件（3月14日及之后）User行**没有**消息级时间戳，这两个字段很多时候和`created`一样 | 如果总大脑认为意义不大，可以只保留`created`，不做`first_message`/`last_message` |
| `models`提取时机 | 应在清洗前提取（因为合并后Agent角色行被改写为`**Cursor (model xxx)**`），还是在改写后从新格式提取？ | 技术上都可以。建议在清洗前提取——原始格式更规范、正则更可靠 |

## 十、消息级时间戳处理评估（选项A vs 选项B）

### 实际查证：时间戳分布情况

| 文件时期 | User行格式 | 时间戳情况 | 文件数 |
|---------|-----------|-----------|--------|
| 3月11日 | `_**User (2026-03-11 14:50Z)**_` | ✅ 有时间戳，但**所有消息时间相同**（=Session创建时间） | 约4个文件 |
| 3月12日-17日 | `_**User**_`（无括号） | ❌ 无消息级时间戳 | 约48个文件 |

**结论：52个文件中，只有约4个有时间戳，且这4个的时间戳都是Session时间（不是真实消息发送时间）。**

### 选项评估

#### 选项A：保留在正文里

格式：`**User** (2026-03-14 10:14Z)` 或 `**User**`（无时间戳时不加）

| 维度 | 评估 |
|------|------|
| **标注助手切片** | 🟡 影响小但不为零。切片按话题切分，行号范围标注。如果角色行格式不统一（有的带时间、有的不带），正则匹配需兼容两种格式 |
| **信号助手提取** | 🟡 信号助手扫描`**User**`做角色分割。带时间戳后格式变为`**User** (xxx)`，正则需更新 |
| **复盘大脑分析** | ✅ 复盘大脑不依赖角色行格式做机械处理，时间信息对"对话节奏分析"有潜在价值 |
| **WF1清洗组装** | 🟡 角色改中文的正则需兼容：`**User**` 和 `**User** (xxx)` |
| **实际价值** | 🔴 **极低**——92%的文件没有时间戳，有的4个文件时间戳还不是真实消息时间（都是Session时间） |

#### 选项B：去掉时间戳，只留 `**User**` / `**Cursor (model xxx)**`

| 维度 | 评估 |
|------|------|
| **标注助手切片** | ✅ 角色行格式统一，正则无需改动 |
| **信号助手提取** | ✅ 扫描`**User**`不变 |
| **复盘大脑分析** | ✅ 不影响 |
| **WF1清洗组装** | ✅ 角色替换逻辑只需处理`**User**`和`**Cursor(?:...)?**`两种（已在指令中更新） |
| **实际价值** | ✅ 正文更干净，格式统一 |

### 对比总结

| 维度 | 选项A（保留时间戳） | 选项B（去掉时间戳） |
|------|-------------------|-------------------|
| 管线兼容 | 需改3处正则（标注助手、信号助手、WF1） | 零改动 |
| 信息价值 | 🔴 极低（92%文件无时间戳，有的也不准） | — |
| 格式一致性 | 🟡 不一致（有的有时间、有的没有） | ✅ 一致 |
| 实现成本 | 中（脚本+管线都要改） | 零 |

### 技术总工评估结论

**建议选B**（去掉消息级时间戳），理由：

1. **数据本身不可靠**——有时间戳的4个文件，时间戳都是Session创建时间而非消息时间，保留了也是误导
