# 清洗车间架构方案

> 数字分身"进料口"：通用内容清洗引擎
> 讨论来源：2026-03-12 Claude + Gemini 多轮推演

---

## 一、定位

**SK-04 的物理载体** — 一个"无情的清洗与分解车间"。

- 不关心内容是业务大单还是朋友闲聊
- 唯一天职：多源脏数据 → 结构化干净部件 → Obsidian 知识库
- **不做建议、不做决策、不做分析**

---

## 二、数据来源（多元）

| 来源 | 入库方式 |
|------|---------|
| 钉钉录音转写 | 导出两个文件（转写+总结）→ 丢进原料收件箱 |
| 微信/社交聊天 | 复制粘贴 → 丢进原料收件箱 |
| AI 对话（Cursor等） | cursor_capture.py 自动写入 |
| 网络视频转写 | 转写后丢进原料收件箱 |
| 任何文本素材 | 手动丢进原料收件箱 |

**入口就是 Obsidian 本身**（手机+电脑都有），不搞机器人、不搞 Webhook。

---

## 三、架构

```
📥 原料收件箱/（统一入口）
        ↓
⚙️ brain/cleaner.py
  ├─ Step 1: 格式统一              【脚本】¥0
  ├─ Step 2: 实体标准化             【脚本】读取 全局实体字典.md 硬替换
  ├─ Step 3: 话题切段+去噪          【便宜模型】Haiku/Flash
  └─ Step 4: 结构化提取             【中等模型】Sonnet/GPT-4o
        ↓
📦 碎片暂存/（待审核）
        ↓
👁️ 审核（在 Obsidian 里完成）
  ├─ 展开原文折叠块逐段比对
  ├─ 不对的直接改/删
  └─ 没问题 → 拖到归档目录
        ↓
📂 归档到对应目录
  ├─ 用户输入/事件记录/
  ├─ 用户输入/人物素材/
  └─ 用户输入/外部资料/
```

## 五、关键组件

### 全局实体字典（数字分身/全局实体字典.md）

人名映射、黑话翻译。清洗脚本每次运行必须先读取。

```markdown
| 可能的叫法 | 标准实体 | 身份 |
|-----------|---------|------|
| 兰心、小张 | 张兰心 | 新会务-执行 |
| 刘哥、老刘 | 刘XX | 新会务-项目总监 |
| 老北京、北京那边 | 北京公司 | 公司实体 |
```

### 原料收件箱（用户输入/原料收件箱/）

所有未经处理的原始素材统一入口。

### 碎片暂存（用户输入/碎片暂存/）

清洗后待人工审核的产出。文件名带时间戳避免冲突。

## 八、工作台 A 实现方案：N8N（Docker 自建，免费）

**选型理由**：多地可用、自动触发、可视化编排、以后易扩展钉钉等数据源。
**部署方式**：本地 Docker 容器（`docker start n8n`），零成本，数据全在自己手里。

### 人工审核方案（基于 Obsidian 原生能力）

**不建新工具**，直接在 Obsidian 里审核：
- 清洗产出自带分段原文（折叠），方便逐段对照
- 不对的段落 → 直接改/删
- 原话有误 → 展开原文复制正确内容替换
- 全部OK → 拖到归档目录

**推荐插件**：
- Parallel Markdown — 左右两栏并排看原文 vs 整理结果
- File Diff — 对比差异可合并（可选）

### 运行方式

```
N8N（Docker 自建，¥0）
  ├── 定时轮询 GitHub 仓库 原料收件箱/ 目录（每 4 小时）
  ├── 检测到新文件 → 触发清洗流程
  ├── 读取 全局实体字典.md → 实体硬替换
  ├── 调 LLM API（GPT-4o-mini）→ 结构化提取
  ├── 生成标准 MD → Git push 到 碎片暂存/
  └── 处理完的原料移到 原料收件箱/已处理/
```

### 核心 Python 逻辑仍然需要

N8N 负责编排和触发，但清洗的核心 prompt 和实体替换逻辑用 Python（N8N 的 Code 节点执行）。

### N8N 节点设计（草案）

1. **Schedule Trigger** — 每 4 小时扫一次
2. **GitHub Node** — 读取 原料收件箱/ 新文件列表
3. **GitHub Node** — 读取 全局实体字典.md
4. **Code Node** — Python 实体硬替换
5. **HTTP Request** — 调 LLM API（结构化提取）
6. **Code Node** — 组装标准 MD（YAML frontmatter）
7. **GitHub Node** — push 到 碎片暂存/
8. **GitHub Node** — 移动原料到 已处理/
