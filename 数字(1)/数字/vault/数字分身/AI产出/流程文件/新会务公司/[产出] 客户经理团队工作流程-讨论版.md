## 〇、上帝视角：这个团队的本质

### 外部知识：客户经理团队的规律

| 规律 | 来源 | 对这个团队的意义 |
|------|------|------------|
| 客户经理团队天然适合自管理模式——每个人独立对接项目，自驱动 | [Kapta - Lead Team of KAMs](https://kapta.com/resources/key-account-management-blog/key-account-management/successfully-lead-team-key-account-managers) | 刘不管人不是致命伤，用机制替代就行 |
| 技术领导 ≠ 人员管理——两个不同的技能，应该拆开 | [lethain.com - Tech Lead Manager Trap](https://lethain.com/tech-lead-managers/) | 刘做执行总控，老板做人员管理 |
| 自管理团队不是没有结构，是结构更清晰——角色、边界、汇报节点明确 | [Corporate Rebels](https://www.corporate-rebels.com/blog/how-to-become-a-self-managing-team) | 越是不靠组长管人，越需要机制清晰 |
| 客户经理能力分层：初级（执行标准流程）→ 中级（独立带项目）→ 高级（统筹+策略） | [数字前站 - CSM能力模型](https://www.shuziqianzhan.com/article/4812.html) | 团队7人分层，匹配不同项目 |
| 客户与经理需要对等匹配——难度高的客户配能力强的人 | [McKinsey - 财富顾问团队](https://www.mckinsey.com.cn/券商财富管理转型之道：如何打造财富顾问团队/) | 项目分配按侧写来，不是按谁空闲 |

### 刘的能力边界（侧写）

| 能力 | 水平 | 工作设计的影响 |
|------|------|---------------|
| 闭环执行 | ✅ 强 | 大项目亲自总控，质量把关 |
| 细节控制 | ✅ 强 | 定义执行标准，1-100填充 |
| 品质坚守 | ✅ 强 | 出品质量底线 |
| 带人做事 | ✅ 能 | 能带出独立做事的人（陈丝巾、破狮已验证） |
| 0-1方案 | ⚠️ 弱 | 不能从零搭框架，需要张/老板给框架她填充 |
| 主动管人 | ❌ 不做 | "你不问我就不会管"——已多次验证 |
| 关心下属 | ❌ 不做 | 不会主动关注组员状态和成长 |
| 向上管理 | ⚠️ 弱 | 有情绪但会执行，需要认可 |
| 沟通协调 | ⚠️ 有限 | 沉默策略、惩罚导向（扣分代替沟通） |

### 团队构成

| 层级 | 人员 | 能力特征 | 适合的项目 |
|------|------|----------|-----------|
| **组长** | 刘 | 执行总控、质量把关、大项目亲自带 | VIP大型项目（500人+） |
| **中级** | 董婷、张张、陈霞 | 有统筹逻辑、能独立带中型项目 | 200-500人项目 |
| **初级+** | 陈丝巾、破狮 | 能独立做具体事、需要思路指导 | 中小型项目执行 |
| **初级** | 哈尼 | 接待执行、物料管理 | 配合角色，不独立带项目 |

> **注**：陈霞的特殊约束——安全感驱动、压力敏感，高压多线程下会失效。适合客户好沟通、节奏可预期的中型项目，不适合高压并行。

### 生态约束

| 约束 | 来源 | 设计影响 |
|------|------|---------|
| 客户关系要沉淀到具体对接人 | 核心安全问题 | 中级组员直接对接客户，关系在她们身上积累 |
| 客户分层不同策略不同 | 生态认知 | 按客户难度匹配组员能力 |
| 刘不管人 | 侧写 | 人员管理归老板，刘只管执行出品 |
| 大项目需要刘亲自把关 | 团队现实 | 刘的价值在VIP项目的品质底线 |

### 核心设计原则

| 原则 | 说明 |
|------|------|
| **刘 = 执行总控，不是人的管理者** | 刘管出品质量，不管人员成长 |
| **老板 = 方向把控者** | 做侧写→给刘结论表，组员成长和考核由老板定，但不直接分配项目 |
| **机制代替管人** | 项目交接模板、进度汇报、客户记录——用制度让团队自转 |
| **能力分层匹配项目** | 按组员能力和客户难度匹配，不是按谁空闲 |
| **自然一点** | 不过度设计，刘现在没管薛那边，问题不大 |

## 一、角色拆分：执行总控 vs 人员管理

**传统模式**：组长 = 管事 + 管人（刘做不到）

**我们的模式**：

```
老板
├── 给方向：做侧写 → 给刘结论表（谁适合什么、不适合什么）
├── 管人：组员成长、考核评估、冲突调解
├── 不直接分配项目（避免组员绕过刘找老板）
│
刘（执行总控 + 项目分配）
├── 管事：大项目亲自总控、质量把关、执行标准制定
├── 分配项目：按老板给的结论表，决定谁做哪个项目
├── 不管人的日常：不做1对1、不做成长计划
│
组员（按层级自转）
├── 中级（董婷、张张、陈霞）：独立带项目、直接对接客户
├── 初级+（陈丝巾、破狮）：跟着中级或刘做项目
└── 初级（哈尼）：接待、物料执行
```

**关键区分**：

| 职责 | 谁做 | 为什么 |
|------|------|--------|
| 项目分配给谁 | 刘（按老板给的结论表） | 老板给指针，刘做分配决定，避免组员绕过刘 |
| 项目执行质量 | 刘 | 她的核心能力 |
| 组员成长规划 | 老板 | 刘不关心人 |
| 项目中遇到问题 | 刘先判断，搞不定找老板 | 刘能解决执行问题 |
| 组员情绪/冲突 | 老板 | 刘的沉默+惩罚风格会恶化冲突 |
| 客户关系维护方向 | 老板定策略，组员执行 | 刘不经营关系 |

## 二、客户-项目-组员匹配逻辑

### 项目分级

| 项目级别 | 规模 | 客户特征 | 负责人 | 配置 |
|----------|------|----------|--------|------|
| **S级** | 500人+/VIP客户 | 高要求、多板块、政治敏感 | 刘亲自带 | 刘 + 2-3个组员 |
| **A级** | 200-500人 | 标准大型项目 | 中级组员独立带 | 董/张张/陈霞 + 1-2个初级 |
| **B级** | 100-200人 | 标准中型项目 | 中级组员独立带 | 1个中级 + 1个初级 |
| **C级** | 100人以下 | 简单项目 | 初级+组员 | 陈丝巾/破狮（刘把关关键节点） |

### 客户-组员匹配原则

| 客户特征 | 匹配组员 | 原因 |
|----------|----------|------|
| 要求高、难伺候 | 刘或董婷 | 抗压能力强、统筹能力强 |
| 正常难度、量大 | 张张 | 能执行多个并行项目 |
| 好沟通、节奏稳 | 陈霞 | 安全感驱动，低压环境发挥好 |
| 简单标准化 | 陈丝巾/破狮 | 按标准流程执行 |
| 配合角色 | 哈尼 | 接待、物料执行 |

> **谁决定分配**：刘。老板做侧写，给刘一张结论表（"陈霞适合XX类客户，不适合YY"），刘按表自己分配。老板不直接指定项目——避免组员绕过刘找老板。

## 三、主线任务（团队整体）

### 主线1：客户关系维护（对接层）

**目标**：成引荐后，客户关系在具体客户经理身上积累。

**流程**：
```
成引荐客户经理给客户
    ↓
客户经理成为客户日常对接人
    ↓
日常需求沟通、进度同步、关系维护
    ↓
项目结束 → 客户经理主动回访
    ↓
记录客户偏好、习惯、注意事项到客户档案
```

**分工**：

| 角色 | 做什么 | 不做什么 |
|------|--------|----------|
| 客户经理（具体组员） | 日常对接、需求收集、进度同步、关系积累 | 高层关系经营（成做） |
| 刘 | S级项目亲自对接、质量把关 | 不负责所有客户的关系维护 |
| 成 | 高层关系、关键节点出面、危机灭火 | 日常不介入 |

**关系沉淀机制**：
- 每个项目结束后，客户经理填写**客户档案更新**（客户偏好、关键人物、注意事项）
- 档案存公司系统，不在个人脑子里
- 客户经理换人时，档案交接

### 主线2：客户需求分析

**目标**：准确理解客户要什么，翻译成内部可执行的需求。

**流程**：
```
客户提出需求（口头/微信/邮件）
    ↓
客户经理梳理 → 需求摘要（用模板）
    ↓
内部对齐：和项目经理组确认可行性、和运营确认合规
    ↓
和客户确认理解是否一致
    ↓
进入方案阶段
```

**需求摘要模板**：
| 字段 | 内容 |
|------|------|
| 客户名称 | |
| 项目名称 | |
| 会议规模 | |
| 时间地点 | |
| 核心需求 | 客户原话 |
| 翻译后需求 | 我们理解的 |
| 特殊要求 | |
| 预算范围 | |
| 关键人物 | 对接人、决策者 |

### 主线3：日程管理（前期主导）

**目标**：前期主导日程安排，会议期间接收变化分发给项目经理组。

**流程**：
```
立项后 → 客户经理收集日程需求
    ↓
整理日程初稿（时间、场地、专家、板块）
    ↓
和客户确认
    ↓
和项目经理组对齐执行细节
    ↓
日程定稿
    ↓
会议期间：接收客户临时变更 → 第一时间分发给项目经理组执行
```

**信息流**：
```
客户 → 客户经理（接收变更）→ 项目经理组（执行变更）
         ↑ 不能反着来 ↑
```

### 主线4：内部资源协调

**目标**：协调公司内部资源配合项目执行。

| 协调对象 | 协调内容 | 频率 |
|----------|----------|------|
| 项目经理组（薛） | 搭建、设备、现场执行 | 每个项目 |
| 运营策划（张） | 策划框架、合规报备、预算 | 每个项目 |
| 设计（周） | 主KV、视觉物料 | 每个项目 |
| 市场拓展（成） | 价格、客户高层关系 | 方案阶段+危机时 |

### 主线5：外部供应商协调

**目标**：对接外部供应商，确保服务到位。

| 供应商类型 | 对接内容 |
|------------|----------|
| 酒店/场地 | 场地预订、餐饮、住宿 |
| 搭建公司 | 配合项目经理组 |
| 物料供应商 | 物料制作、运输 |
| 其他 | 按项目需求 |

### 主线6：接待统筹

**目标**：现场接待总控。

**流程**：
```
会前2周 → 制定接待方案
    ↓
分工：谁接机、谁签到、谁引导、谁陪同
    ↓
接待物料准备
    ↓
会前1天 → 接待组内部对齐
    ↓
现场执行 → 突发应变
    ↓
会后 → 送客、反馈收集
```

### 主线7：方案整合与客户沟通

**目标**：整合各模块方案，和客户沟通确认。

**分工**（已确认）：

| 模块 | 谁出 | 客户经理做什么 |
|------|------|--------------|
| 策划框架 | 张/老板 | 整合到方案中 |
| 总体价格 | 成 | 整合到方案中 |
| 接待+宣传 | 刘/客户经理 | 自己做 |
| 搭建设备 | 薛 | 整合到方案中 |
| 主KV/视觉 | 周 | 整合到方案中 |

```
各模块分别出内容
    ↓
客户经理整合成完整方案
    ↓
和客户沟通、确认、修改
    ↓
定稿 → 内部分发执行
```

## 四、刘的专属职责（执行总控）

| 职责 | 做什么 | 频率 |
|------|--------|------|
| **项目分配** | 按老板给的结论表，决定谁负责哪个项目 | 每个新项目 |
| **S级项目亲自总控** | 全程跟进、质量把关、现场主控 | 按项目 |
| **执行标准制定** | 定义接待标准、物料标准、客户沟通标准 | 持续更新 |
| **关键节点把关** | A/B级项目在关键节点（方案定稿、现场前1天）检查 | 按项目 |
| **异常升级** | 组员搞不定的执行问题，刘先判断、先处理 | 按需 |
| **新人带教** | 新入职的初级成员，带着做1-2个项目 | 新人期 |

**刘不做的事**：

| 不做 | 谁做 |
|------|------|
| 组员成长计划 | 老板 |
| 组员情绪/冲突 | 老板 |
| 周报审核 | 运营（张）追踪进度 |
| 客户关系策略 | 成/老板 |

## 六、按项目阶段的完整流程

### 1. 立项阶段

| 步骤 | 谁做 | 产出 |
|------|------|------|
| 成引荐客户经理给客户 | 成 | 客户经理和客户建立联系 |
| 了解客户初步需求 | 客户经理 | 需求摘要 |
| 客户重点分析 | 客户经理（+成补充高层信息） | 客户重点清单 |
| 项目分级（S/A/B/C） | 刘（按结论表） | 确定负责人+配置 |
| 内部立项 | 客户经理 | 立项通知 |

### 2. 方案阶段

| 步骤 | 谁做 | 产出 |
|------|------|------|
| 收集各模块方案 | 客户经理 | 各模块草稿 |
| 整合成完整方案 | 客户经理（S级刘把关） | 完整方案 |
| 客户沟通确认 | 客户经理 | 修改意见 |
| 方案定稿 | 客户经理 | 定稿方案 |
| 关键节点：S/A级项目定稿前刘检查 | 刘 | 质量确认 |

### 3. 执行准备

| 步骤 | 谁做 | 产出 |
|------|------|------|
| 内部资源确认 | 客户经理 | 资源确认表 |
| 外部供应商确认 | 客户经理 | 供应商确认 |
| 日程定稿 | 客户经理 | 总日程表 |
| 接待方案 | 客户经理 | 接待分工表 |
| 物料清单 | 客户经理 + 初级组员 | 物料清单 |
| 关键节点：刘检查执行准备 | 刘 | 执行准备确认 |

### 4. 现场执行

| 步骤 | 谁做 | 说明 |
|------|------|------|
| 接待总控 | 客户经理（S级刘在现场） | 签到、引导、陪同 |
| 客户变更接收 | 客户经理 | 第一时间响应 |
| 分发给项目经理组 | 客户经理 | 执行侧落地 |
| 突发处理 | 客户经理（搞不定找刘/成） | 现场应变 |

### 5. 收尾阶段

| 步骤 | 谁做 | 产出 |
|------|------|------|
| 客户反馈收集 | 客户经理 | 反馈记录 |
| 客户档案更新 | 客户经理 | 客户偏好、关键人物、注意事项 |
| 项目文档归档 | 客户经理 | 完整项目档案 |
| 结算配合 | 客户经理 + 运营 | 结算文件 |
| 复盘记录 | 客户经理 | 复盘报告 |
| 客户关系维护 | 客户经理 | 主动回访 |

## 七、每周时间分配（工作量评估）

### 单人工作量（中级组员，独立带1个A级项目）

#### 正常周（项目准备期）

| 任务类型 | 时间/周 | 说明 |
|----------|---------|------|
| 客户日常沟通 | 5-8小时 | 需求对接、进度同步 |
| 日程管理 | 3-5小时 | 收集、整理、确认 |
| 内部协调 | 3-4小时 | 和各组对齐 |
| 供应商对接 | 2-3小时 | 场地、物料 |
| 方案整合 | 3-5小时 | 整合各模块 |
| 文档/行政 | 2-3小时 | 记录、归档 |
| **合计** | **18-28小时** | 约3-4天 |

#### 高峰周（会议前1周+会议期间）

| 任务类型 | 时间/周 | 说明 |
|----------|---------|------|
| 客户密集沟通 | 10-15小时 | 变更不断 |
| 接待统筹 | 8-12小时 | 现场全程 |
| 内部协调 | 5-8小时 | 各种对齐 |
| 供应商 | 3-5小时 | 临时调整 |
| 突发处理 | 3-5小时 | 不可预测 |

|------|-------------|-----------|----------------|
| 刘 | S级1个 + 把关A级 | S级1个（全力投入） | 3-5个S级 |

## 九、风险控制机制（6项）

### 1. 项目分配机制（对抗"按谁空闲分配"）

**问题**：如果随便分配，压力敏感的人接到难客户会崩。

**机制**：
```
老板做侧写 → 给刘一张结论表（谁适合什么、不适合什么、并行上限）
刘按结论表分配 → 组员认为是刘安排的，不会绕过刘找老板
```

**结论表格式**（给刘的）：

| 人 | 适合 | 不适合 | 并行上限 |
|----|------|--------|---------|
| 董婷 | 难客户、大项目、多线程 | — | 2个A级 |
| 张张 | 标准项目、量大 | — | 2个A级 |
| 陈霞 | 好沟通客户、节奏稳 | 难伺候的客户、多个并行 | 1个A级 |
| 陈丝巾 | 中小型、有明确流程的 | 需要统筹多板块的 | 2个B级 |
| 破狮 | 同上 | 同上 | 2个B级 |
| 哈尼 | 配合角色 | 不独立带项目 | 跟项目走 |

**执行**：刘每次有新项目时按表分配。老板偶尔抽查分配是否合理。

> **为什么不让老板直接分配**：老板分配→组员的心理归因是"老板安排我的"→遇到问题找老板或成→刘被架空。刘分配→归因在刘→问题先到刘。

---

### 2. 客户档案沉淀机制（对抗关系在个人脑子里）

**问题**：客户经理走了，客户信息也走了。

**机制**：
```
每个项目结束后 → 客户经理更新客户档案
档案内容：偏好、关键人物、注意事项、历史问题
存在公司系统 → 换人时可交接
```

**检查**：运营每月检查档案更新情况。

---

### 3. 关键节点把关机制（对抗质量不稳定）

**问题**：中级/初级独立带项目，出品质量可能不稳定。

**机制**：
```
S级项目 → 刘全程把关
A级项目 → 方案定稿前 + 现场前1天，刘检查
B级项目 → 方案定稿前，刘快速过一遍
C级项目 → 不需要刘把关
```

**执行**：刘在关键节点检查，不全程介入。

---

### 4. 异常升级机制（对抗组员搞不定的问题）

**问题**：组员遇到搞不定的客户问题或执行问题。

**机制**：
```
组员搞不定 → 先找刘
刘搞不定或涉及关系/商务 → 找成或老板
涉及赔偿/重大让步 → 必须老板授权
```

---

### 5. 进度可视化机制（对抗信息黑洞）

**问题**：刘不主动汇报，组员也不主动。

**机制**：
```
每个项目 → 运营（张）建立进度表
客户经理每周更新进度
异常 → 立即通知
```

**执行**：运营追踪进度，不依赖刘主动管理。

---

### 6. 复盘沉淀机制（对抗同样的错重复犯）

**问题**：做完就做完了，经验没沉淀。

**机制**：
```
每个重要项目结束后 → 客户经理写复盘记录
运营组织复盘会（重要项目）
复盘内容：做好了什么、出什么问题了、下次怎么改
```

## 十三、总结

### 这个团队的本质

```
客户经理团队 = 客户对接的主力军
        ↓
    刘 = 执行总控+质量把关+项目分配（按结论表）
        ↓
    中级组员 = 独立带项目、直接对客户、关系在她们身上积累
        ↓
    初级组员 = 跟着做项目、在项目中成长
        ↓
    机制 = 代替"组长管人"——项目分配靠侧写、进度靠运营追踪、质量靠关键节点把关
```

### 设计逻辑

| 设计 | 理由 |
|------|------|
| 刘做执行总控+项目分配 | 发挥闭环执行能力，按结论表分配项目 |
| 老板做侧写+管人 | 给刘结论表（不直接分配），管组员成长和冲突 |
| 能力分层匹配项目 | 客户-项目-组员对等 |
| 机制代替管人 | 客户档案、进度追踪、关键节点把关、复盘 |
| 关系沉淀到组员 | 成引荐后退居二线，客户经理直接积累 |

### 关键成功因素

1. **结论表简单到刘能直接用** —— 否则刘会忽略，按"谁空闲"分
2. **客户档案真的在更新** —— 否则关系在个人脑子里
3. **运营追踪进度** —— 否则信息黑洞
4. **刘在关键节点真的看** —— 否则质量不稳定
5. **陈霞的项目不要压太多** —— 结论表已写明上限，刘按表执行
