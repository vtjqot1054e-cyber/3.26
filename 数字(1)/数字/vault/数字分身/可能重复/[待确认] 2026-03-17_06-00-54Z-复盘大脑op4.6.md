#### 2.2 AI（本次对话中的行为）

##### ① 碎片整合执行质量——本篇表现好

**为什么本篇不跳？** 因为本篇的用户输入全是策划型内容（架构设计），不是执行指令。AI在面对"帮我装N8N"时跳过碎片整合，在面对"帮我策划窗口体系"时不跳。

**结论**：AI的碎片整合不是"会不会"的问题，而是**"什么场景触发"的问题**。收到执行指令→直接做；收到策划需求→做碎片整合。前者是违规的。

##### ② 上帝视角——本篇出现了前三篇没有的主动干预

**为什么只在本篇出现？** 可能因为用户在第1次问完后岔到账号问题又第2次重复问——重复行为比漂移更容易被AI捕捉到。前两篇是"漂移到新话题"，AI没有检测机制；本篇是"重复同一个问题"，AI的去重逻辑触发了。

**结论**：AI目前的"上帝视角"只能识别**重复**，不能识别**漂移**。这是一个系统性缺陷。

##### ③ 框架初版偏浅

- v1把大脑当"聊天窗口"→ 被用户纠正
- v2没想到"胜率导师独立"→ 用户补充

**可写入系统元提示词的建议**：

### 一、切片标准：怎么切？

**按话题转折切**，不是按用户发言或时间间隔。理由：

- 一个"话题"可能包含多轮用户+AI对话（比如"讨论窗口体系v1→用户纠正→AI修正v2"是同一个话题的三轮）
- 用户的发散是按话题跳的，不是按时间跳的
- 切片的目的是让用户审核"一个完整的话题"的价值，而不是审核零散的段落

**切片判断规则**：

| 信号 | 怎么切 |
|------|--------|
| 用户换了一个新问题/新需求 | 新切片起点 |
| 话题从A完全跳到B（无过渡） | 新切片起点 |
| AI做了长篇回复后用户继续追问同一话题 | 同一切片 |
| 用户说"接着做下一件事" | 新切片起点 |
| 启动仪式/工具日志 | 独立切片（通常整片标🗑） |

**切片粒度目标**：一篇1500行的对话切成15-25个切片，每个切片50-150行。用户审核的是切片摘要表，不是原文。

---

### 二、切片摘要的语言风格

**用用户的词，不用AI修辞。** 具体规则：

| 做 | 不做 |
|---|---|
| "用户问多窗口能不能关联" | "用户探讨了多窗口协同的可能性" |
| "AI说做不到，给了4个替代方案" | "AI提供了全面的替代方案分析" |
| "用户纠正：大脑不是聊天窗口" | "用户对大脑定位提出了深刻的修正意见" |
| "跑偏到账号充值问题" | "用户临时切换到了订阅管理话题" |

摘要控制在一句话，最多两句。用户一眼看完20个切片摘要，就知道这篇对话的全貌。

---

### 三、切片格式（替代逐段清单）

**切片模式替代逐段清单**，不并存。理由：逐段清单30行，用户看不完；切片摘要20行，用户能一次审完。

**新格式：**

```markdown
### 切片审阅表

| # | 话题 | 行范围 | 摘要 | AI判断 | 理由 | 与现状关系 | 用户终审 | 用户理由 |
|---|------|--------|------|--------|------|-----------|---------|---------|
| 1 | 启动接续 | 68-121 | 用户说接上文，AI读文件总结待办 | ❌ | 每篇都有的启动仪式 | 已过时（待办已完成） | ⏳ | |
| 2 | 多窗口关联调研 | 126-308 | 用户问Cursor能不能智能关联窗口，AI调研确认不能 | ✅ | 奠定"自建体系"方向 | 现在用的就是自建体系 | ⏳ | |
| 3 | 账号充值岔路 | 312-498 | 用户问换账号充值/多注册QQ邮箱，AI逐条回答 | ❌ | 与主线无关 | 已用Ultra200解决 | ⏳ | |
| ... | | | | | | | | |
```

**列说明：**
- **话题**：2-6个字的话题名
- **行范围**：原文起止行号，用户想看细节可以定位
- **摘要**：一句话，用用户的词
- **AI判断**：✅有价值 / ❌无效 / ❓不确定
- **理由**：为什么这样判断
- **与现状关系**：这个话题现在还有没有用（关键列——帮用户快速判断"旧观点现在还成立吗"）
- **用户终审**：⏳/✅/❌/❓
- **用户理由**：用户为什么这样标（特别是❓必须记录）

---

### 四、"与现状关系"列——这是切片模式的核心创新

这一列直接解决"新→旧审核顺序"的配套问题。用户先审了最新的对话，建立了当前认知锚点，再回头审旧对话时：

- **已过时**：这个决策/观点后来被推翻了（比如"SpecStory是核心机制"→后来废弃了）
- **已验证**：这个决策后来验证是对的（比如"自建窗口体系"→现在在用）
- **已演化**：观点没错但后来升级了（比如"7窗口方案"→后来变成v3）
- **仍有效**：到现在还成立
- **不确定**：无法判断

标注助手做这个判断需要读交接备忘和已有存档——这和信号助手读存档做跨篇验证是同一个动作，但标注助手只需要判断"与现状关系"，不需要做深度分析。

---

### 五、信号助手怎么适配切片模式

信号助手的输入从"逐段清单"变成"切片审阅表"。适配方式：

| 原来 | 现在 |
|------|------|
| 逐段对比AI标记vs用户标记 | **逐切片对比AI判断vs用户终审** |
| 画像信号从单段提取 | **画像信号从切片维度提取（一个切片可能包含多个信号）** |
| 用户否决=单段 | **用户否决=整个切片的判断被翻转** |

信号助手的对比维度新增：

| AI判断 | 用户终审 | 信号 |
|--------|---------|------|
| ✅ | ❌ | 🔴 AI高估——用户认为整个话题无价值 |
| ❌ | ✅ | 🔴 AI低估——用户认为被标垃圾的话题有价值 |
| ✅ | ❓ | 🟡 用户不确定——需要追踪验证 |
| ❌ | ❓ | 🟡 同上 |
| 任何 | 用户改了"与现状关系" | 🔴 标注助手对现状判断有误 |
