## 结论先说：Cursor 目前做不到你要的“智能控制和关联”

| 你想要的              | Cursor 实际能力                     |
| ----------------- | ------------------------------- |
| 多窗口各用不同模型         | 同项目下基本是单窗口，多的是**标签（tabs）**，不是窗口 |
| 每窗口/标签绑定不同模型      | ✅ 支持，per-tab 模型选择（约 2025.8）     |
| Agent 能“智能控制”多个窗口 | ❌ 不支持，每个 Agent 彼此独立             |
| 自动告诉你“当前问题该去哪个窗口” | ❌ 不支持，没有这种元 Agent               |

---

## 具体限制

1. **同一项目多窗口**  
   Cursor 会优先回到已有窗口，很难真正实现你说的 A/B/C/E 四个独立窗口。

2. **多标签 + 不同模型**  
   可以实现：  
   - 标签 A → Opus 4.6  
   - 标签 B → Sonnet 4.6  
   - 标签 C → Gemini 3.1 Pro  
   - 标签 E → Auto  

3. **跨标签/跨窗口的“智能关联”**  
   Cursor 没有这类能力。每个 Agent 标签是孤立的，没有：
   - 跨标签共享上下文  
   - 一个“总控”把任务路由到某个模型  
   - “当前问题适合哪个窗口/模型”的自动推荐  

所以，“智能控制和关联这些窗口并告诉你该去哪个窗口”在 Cursor 里目前**无法自动完成**。

---

## 可行的替代做法

[^1]**方案 A：手动分工（当前唯一靠谱做法）**  
你事先定好：  
- 复杂推理 / 架构 → Opus  
- 常规任务 / 写码 → Sonnet  
- 多模态 / 长上下文 → Gemini  
- 日常小问题 → Auto  

然后根据当前问题类型，**自己选标签**。

[^2]**方案 B：数字分身补一层“顾问式推荐”**  
在数字分身里（例如 SK-06 上帝视角 + 碎片 SK）做一层逻辑：  
- 你在任意一个窗口描述问题  
- AI 分析后给出建议：  
  - “这类问题更适合用 Sonnet，建议切到 B 标签”  
  - “需要深度推理，更适合用 Opus，建议切到 A 标签”  

这是**人工决策 + 建议**，不是 Cursor 自动路由，但能起到“该去哪聊”的指引作用。

## 一、Cursor 官方插件市场（cursor.com/marketplace）

### 1. Agent Orchestration 类

| 插件 | 能力 | 和你需求的匹配度 |
|------|------|------------------|
| **Runlayer** | MCP 安全治理：审计、策略、密钥保护 | ❌ 偏向安全，不做模型/窗口推荐 |
| **Compound Engineering** | 20+ 专业 Agent（代码审查、规划等），跨 Cursor/Claude Code/Copilot | ⚠️ 多 Agent 分工，但**不负责**「该去哪个窗口」 |
| **Langfuse** | 追踪、评估 LLM 调用 | ❌ 偏观测，不做路由 |
| **Parallel** | 并行搜索、研究、数据增强 | ❌ 能力不同 |

### 2. 其他相关

| 插件 | 说明 |
|------|------|
| **Continual Learning** | 从 transcript 学习偏好，维护 AGENTS.md |
| **Cursor Team Kit** | 官方 CI、代码审查、PR 流程 |

结论：官方市场**没有**「按问题类型推荐去某个窗口/模型」的插件。

## 二、GitHub 开源（和需求最接近的）

### 1. madebyaris/agent-orchestration ⭐4

- **仓库**: https://github.com/madebyaris/agent-orchestration
- **形式**: MCP Server
- **能力**: 多 Agent 协作：共享内存、任务队列、资源锁、Cursor rules、AGENTS.md 工作流
- **解决的问题**: 竞态、重复工作、协调多 Agent（Cursor / Aider / Windsurf / Copilot）

**匹配度**: ⚠️ 做的是「多个 Agent 如何协作」，不是「该用哪个窗口/模型」；但能作为多 Agent 协调的基础设施。

---

### 2. thiswind/cursor-agent-team ⭐21

- **仓库**: https://github.com/thiswind/cursor-agent-team
- **形式**: 单会话内的多角色协作（架构师 / 工程师 / 审查员）
- **能力**: 在一个对话里切换角色、协作完成任务

**匹配度**: ❌ 单会话多角色，不涉及多窗口、多模型选择。

---

### 3. alexhatzo/agent-squared ⭐2

- **仓库**: https://github.com/alexhatzo/agent-squared
- **形式**: MCP，Agent 的 Agent
- **能力**: 任务分解 → Prompt 工程 → 专家 Agent（前端/后端/云/安全）→ Composer → 结果验证

**匹配度**: ⚠️ 任务路由到**不同专业 Agent**，而不是路由到**不同窗口或模型**；可作为「路由逻辑」的参考实现。

---

### 4. pycursor_agent

- **说明**: Python 封装 Cursor Agent CLI，支持多模型（Gemini、Opus、GPT 等）统一调用
- **用途**: 外部脚本/自动化调用不同模型，不是 IDE 内多窗口推荐

---

## 三、结论与建议

| 需求 | 现状 |
|------|------|
| 多标签不同模型（Opus / Sonnet / Gemini / Auto） | ✅ 原生支持，无需插件 |
| 「总控」判断当前问题适合哪个模型 | ❌ 无现成插件 |
| 自动/半自动推荐「该去哪个窗口/标签」 | ❌ 无现成插件 |

==[AI亮]**总体结论**：目前没有插件能做到「智能控制多窗口 + 告诉你在哪个窗口对话」。  ==
官方市场偏 MCP 治理、多 Agent 工作流；GitHub 上以多 Agent 协作、任务分解为主，没有「窗口/模型推荐」类实现。

**可选方案**：

1. **手动分工**：在数字分身里固化规则（如 SK-06 / 能力栈）：  
   - 复杂推理/架构 → Opus  
   - 常规写码/调试 → Sonnet  
   - 长上下文/多模态 → Gemini  
   - 日常小问题 → Auto  

2. **自建「顾问式推荐」**：用项目内的 SK/规则（例如 SK-06 上帝视角），在任意一个窗口提问时，由 AI 给出「建议用 XX 模型 / XX 标签」。实现成本低、可迭代。

3. **跟踪 agent-orchestration / agent-squared**：如果之后想尝试「多 Agent 自动分工」，可在此基础上做二次开发，把模型选择/窗口建议接进去。

需要的话，我可以帮你把「顾问式推荐」写成一条可放进 `.cursor/skills/` 或交接备忘里的具体规则草案。

### 窗口交付机制

每个窗口关闭前，必须完成三个方向的交付：

```
                    ┌─────────────┐
                    │  上级交付     │
                    │ （给分大脑/   │
                    │  总大脑）     │
                    └──────┬──────┘
                           │
┌──────────────┐     ┌─────┴──────┐     ┌──────────────┐
│ 关联窗口交付  │◄────│  本窗口     │────►│  续窗口交付   │
│（影响其他线   │     │  关闭前     │     │（下一个同类   │
│ 的结论）      │     └────────────┘     │  窗口需要的） │
└──────────────┘                        └──────────────┘
```

| 交付方向 | 交给谁 | 内容 | 写在哪 |
|----------|--------|------|--------|
| **上级交付** | 分大脑 / 总大脑 | 本次结论、状态变化、需要上级决策的事 | 交接备忘.md |
| **续窗口交付** | 自己的下一个同名窗口 | 断点、待办、上下文摘要 | 交接备忘.md + SpecStory |
| **关联窗口交付** | 其他线的窗口 | 影响其他线的结论 | 对应线的待办区 / 交接备忘.md |

结论：AI目前的"上帝视角"只能识别重复，不能识别漂移。这是一个系统性缺陷。

##### ③ 框架初版偏浅

- v1把大脑当"聊天窗口"→ 被用户纠正
- v2没想到"胜率导师独立"→ 用户补充
- AI在每次被纠正后的响应质量很高（快速落地、结构清晰），但初版的思考深度不够

可写入系统元提示词的建议：

> AI在提出架构方案时，必须先自问：这个方案把每个组件当成了什么？是工具还是有自主性的实体？如果用户用"操作系统""指挥官"等词描述过某个组件，初版方案必须从那个层级起步，不能降级到"聊天窗口"。
