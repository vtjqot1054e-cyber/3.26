### 🤖 Cursor

不需要。Docker 和同步是两回事。

**Docker 只是 N8N 的运行方式**，跟两台机器之间能不能同步没有关系。类比：Word 装在 C 盘还是 D 盘，不影响你用 U 盘拷文件。

两台机器的 N8N 同步靠的是这个链条：

```
公司 N8N（Docker）
    → 导出/修改工作流 JSON → 存在 Git 仓库 brain/n8n_scripts/
    → git push

家里 N8N（npm）
    → git pull → 导入同一个 JSON
    → 同样的工作流跑起来
```

**真正不同步的是凭据**（GitHub token、OpenAI API key）——这些存在每台机器本地的 N8N 数据库里，不管 Docker 还是 npm 都一样要各配一次。这是安全设计，不是 Docker 的问题。

| 项目 | 能同步吗 | 靠什么同步 |
|------|---------|-----------|
| 工作流（节点、逻辑、代码） | 能 | Git + JSON 文件 |
| 凭据（API key 等） | 不能 | 各机器本地配 |
| 执行历史 | 不能 | 各机器本地存 |

**所以现在的 npm 安装完全没问题。** 你在公司 Docker 里调好的工作流，JSON 已经在 Git 仓库里了（`n8n_dialogue_review_workflow.json`、`n8n_cleaner_workflow.json`），家里导入就能用。

## 可写入核心画像 / 系统元提示词的模式（用户提出的需求）

以下两条模式在本次对话中有明确证据，且用户判断若多次出现需固化：

模式 A（用户侧）： 用户在执行层会迁就 AI 建议但不主动维护执行约束 → AI 需要在"用户认可某个方向"之后，主动提出约束："你刚才同意了先只做 X，我要开始做 X 之前，再确认一次：今晚只做 X 对吗？"

模式 B（AI 侧）： 用户给出执行指令时，AI 的默认行为是"理解 → 执行"，缺失"理解 → 碎片整合 → 确认目标对齐 → 执行"这个中间步骤 → 系统元提示词层面需强制触发碎片整合，不能由 AI 自由判断"这条指令是否需要先确认"。
