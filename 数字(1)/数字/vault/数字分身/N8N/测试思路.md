## 工作流1：01-对话清洗标注

### 导入步骤

1. N8N → Workflows → Import from File → 选 `E:\----2\N8N\01-对话清洗标注.json`
2. 打开后检查所有 GitHub 节点的凭据是否自动绑定
3. 如未绑定，手动在每个 GitHub 节点选择凭据

### 各节点验证清单

| 节点 | 验证方法 | 预期结果 |
|------|---------|---------|
| 读取实体字典 | 单独执行，查看 Output | `content` 字段有 base64 字符串（或二进制数据） |
| 列出对话导出 | 单独执行 | 返回 `数字分身/对话导出/` 目录下所有文件列表 |
| 列出待确认 | 单独执行 | 返回 `数字分身/待确认/` 下文件（首次运行应只有 .gitkeep） |
| 过滤未处理 | 执行到此 | 只输出对话导出有、待确认没有的文件；已处理文件不重复 |
| 读取原文 | 执行到此 | `$json.content` 有值（base64 编码的 MD 文件内容） |
| 清洗组装 | 执行到此 | `outputContent` 字段非空，含 YAML frontmatter + 正文；`replaced_count` >= 0 |
| 写入待确认 | 执行完整流 | GitHub 上 `数字分身/待确认/[待确认] xxx.md` 文件存在且有内容 |

### 关键检查点

1. **base64 解码**：进 清洗组装 节点的 Input，检查 `$json.content` 是 base64 字符串（以 `A` 开头、含 `=` padding）还是已空（则走 `$binary` 分支）
2. **字典替换**：如果实体字典有内容，输出中应能看到 `~~别名~~==标准名==` 格式的标记
3. **格式转换**：输出的正文中应看到 `### 👤 用户 [HH:MM]` 和 `### 🤖 模型名 [mode]` 格式

### 常见问题

- **列出待确认失败（404）**：`数字分身/待确认/` 目录不存在 → 需要先在 GitHub 创建该目录
- **As Binary Property 问题**：若 `$json.content` 为空，确认 GitHub 节点未启用"As Binary Property"；或代码会自动 fallback 到 `$binary.data.data`
- **字典为空**：`数字分身/全局实体字典.md` 文件不存在或格式不符合 `| 别名 | 标准名 |` 表格格式

## 工作流2：02-用户校对提炼

### 使用前提

1. 工作流1已经跑出了至少一个 `[待确认] xxx.md` 文件
2. 用户在 Obsidian 中打开 `数字分身/待确认/` 目录
3. 对 `[待确认] xxx.md` 进行校对：
   - 接受替换：删掉 `~~原文~~` 保留 `==标准名==`，或者直接删成纯文本标准名
   - 拒绝替换：把 `~~原文~~==标准名==` 整段删掉，恢复原文
4. 保存并 push 到 GitHub

### 各节点验证清单

| 节点 | 验证方法 | 预期结果 |
|------|---------|---------|
| 列出待确认 | 执行到此 | 返回 `数字分身/待确认/` 下文件列表 |
| 列出对话存档 | 执行到此 | 返回 `数字分身/对话存档/` 下文件（首次为空） |
| 过滤待处理 | 执行到此 | 输出待确认中尚未存档的文件；每个 item 含 `pendingPath`、`originalPath` |
| 读取用户改版 | 执行到此 | `$json.content` 有值（用户校对后的版本） |
| 读取原始文件 | 执行到此 | `$json.content` 有值（对话导出的原始版本） |
| 对比提炼 | 执行到此 | `archiveContent` 和 `signalContent` 均非空 |
| 写入对话存档 | 执行完整流 | GitHub 上 `数字分身/对话存档/[存档] xxx.md` 有内容，YAML中 `status: confirmed` |
| 写入信号报告 | 执行完整流 | GitHub 上 `数字分身/对话存档/信号/[信号] xxx.md` 有替换结果表格 |

### 关键检查点

1. **过滤逻辑**：`过滤待处理` 节点的 item 中应有 `originalPath = 数字分身/对话导出/xxx.md`，核对路径正确
2. **原始文件路径**：`读取原始文件` 的 filePath 表达式 `$('过滤待处理').item.json.originalPath` 应能正确引用
3. **存档 YAML**：检查 `[存档]` 文件中 `type: 对话存档`、`status: confirmed`、`confirmed_date` 字段是否正确

## 快速 Smoke Test（最小验证）

1. 导入 WF1，手动触发，只跑"读取实体字典 → 列出对话导出 → 列出待确认"这三步（在第4节点前打断点）
2. 确认三个节点都返回数据
3. 继续执行全流，检查 GitHub 上是否出现 `[待确认]` 文件且有内容
4. 导入 WF2，做最小校对（接受一个替换），手动触发，检查 `[存档]` 和 `[信号]` 文件
