## 诊断：卡顿的真正原因

你的卡顿不是因为"监控方案落后"，而是两个脚本在 **竞争 Cursor 的文件 I/O**：

**`cursor_capture.py` 的问题：**
1. **每 2 秒全量读取 JSONL 文件** — `jf.read_text()` 把整个文件读进内存，只是按行数跳过已处理的行，但 I/O 负担是全量的
2. **遍历所有项目目录** — `CURSOR_PROJ.iterdir()` + `rglob("*.jsonl")` 每 2 秒扫一次磁盘
3. **与 Cursor 写入竞争** — Cursor 正在往 `agent-transcripts/*.jsonl` 写数据，你的脚本同时在读，Windows 文件锁机制下会互相阻塞

**`对话监控.pyw` 的问题：**
4. **也是每 2 秒轮询** — `tick()` 每 2 秒检查 mtime + 可能触发全量重渲染
5. **tkinter 全量重绘** — 每次文件变化，`render()` 清空整个 Text widget 再重建，大文件下极其缓慢

**叠加效应**：两个脚本 + Cursor 本身 = 三方同时争抢磁盘 I/O，每 2 秒一个周期。

## 意见一："采集器不能搞复杂了"

TABLE date as "日期"
FROM "用户输入/碎片暂存"
WHERE status = "done"
SORT file.mtime DESC
LIMIT 10

## 关键发现：有三类素材进不了管道

**文字类（6种）**：复制粘贴到 Obsidian → 保存到收件箱 → N8N 处理。**通路通畅。**

**非文字类（3种）**：图片、表格、文档。这些有**断点**：

```
图片 → 丢到收件箱 → N8N 读到的是文件路径，不是图片内容 → 💀
Excel → 丢到收件箱 → N8N 读到的是二进制乱码 → 💀
PDF   → 丢到收件箱 → 同上 → 💀
```

## 非文字素材怎么办？

每种有不同的最简解法：

| 素材 | 最简解法 | 成本 |
|------|---------|------|
| **图片** | 你拍完/截完图后，丢给 Gemini 说"帮我把这个转成文字" → 复制结果到收件箱 | 0（你已经在用 Gemini） |
| **表格** | 在 Excel 里选中 → 复制 → 粘贴到 Obsidian（自动变成 Markdown 表格） | 0 |
| **PDF/Word** | 打开 → 复制文字部分 → 粘贴到 Obsidian | 0（绝大多数场景够用） |

**共同点：人肉转成文字，粘贴到 Obsidian。** 不搞自动化，因为：
1. 你的量不大（不是每天处理100个PDF）
2. 人肉复制 = 顺便过一遍脑子，本身就是轻度审核
3. 自动化 OCR/PDF解析 引入新依赖，违反"先简单"原则

**Cursor**

你说得对，**这不是我们要的。**

SpecStory 存的是**完整的技术日志**，不是干净对话：

- `<think>` 块：AI 的内部思考（英文，大段）
- `<tool-use>` 块：每次读文件、写文件的记录
- HTML 标签到处都是

对比一下：

| | Export Transcript（手动导出） | SpecStory（自动保存） |
|--|--|--|
| 用户消息 | 干净 | 干净 |
| AI 回复 | 只有最终回复 | 包含思考过程 + 工具调用 |
| 噪音 | 几乎没有 | 大量 |
| 你之前导出那个 | 285 行 | 同一个对话 698 行 |

**Export Transcript 是你要的那个"照片"，SpecStory 是"监控录像含后台画面"。**

所以回到现实：

1. **SpecStory 可以留着当备份**（完整记录，万一需要追溯工具操作细节）
2. **原料来源还是用 Export Transcript**——干净，就是你看到的对话
3. 唯一的缺点是手动点一下，但**一个话题一个窗口 + 结束时点一下导出**，操作量很小

或者：N8N 读 SpecStory 文件时加一步轻清洗（去掉 `<think>` 和 `<tool-use>` 块），自动化就保住了。这个清洗很简单，几行正则就搞定。
