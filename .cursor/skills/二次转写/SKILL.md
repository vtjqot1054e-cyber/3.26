# 二次转写 SK v1.2

## 🚫 禁止行为（必须遵守）

| 禁止 | 说明 |
|------|------|
| **禁止创建新文件** | 必须在用户指定的原文件中修改，不得生成 `-二次转写.md` 等新文件 |
| **禁止总结/精炼** | 不做内容总结、不做段落精炼、不做要点提取 |
| **禁止删除原文** | 原文一个字都不能删，只能在原文下方添加转写 |
| **禁止合并段落** | 每个发言人段落单独转写，不得把多段合成一段 |
| **禁止输出到对话框** | 转写内容必须直接写入文件，不是发到对话框让用户复制 |

---

## 一、定位

| 维度 | 说明 |
|------|------|
| **本质** | 口语 → 书面语的质量转换工具 |
| **输入** | ASR 转写的口语文字（会议录音、访谈等） |
| **输出** | 原文 + 语义转写（上下对照格式，直接写入原文件） |
| **下游** | 用户校准 → 数字真我复盘 → 入库 |

---

## 二、工作模式

### 核心流程

```
1. 用户提供文件路径 + 参与者识别（发言人1=谁、发言人2=谁...）
2. AI 读取文件，分段处理（每段约 50-100 行）
3. AI 直接在原文件中：
   - 添加 YAML frontmatter（首次）
   - 在每个发言人段落下方添加「**语义转写**：」
   - 用 `---` 分隔不同发言段
4. 用户在文件中校准（改错、补漏）
5. 用户说"继续"，AI 处理下一段
6. 循环直到全文完成
```

### 为什么分段？

1. **节约 Token**：避免一次性发送 1000 行
2. **质量可控**：小段落更容易发现转写错误
3. **人工校准**：用户边看边改，不是事后批量修

### 为什么直接写入原文件？

1. **方便对照**：原文和转写上下紧邻，一眼看出偏差
2. **避免来回复制**：AI 直接改文件，用户直接在文件里校准
3. **保留原文**：原文不删除，校准后可追溯

## 三、输出格式

### 文件头部（首次处理时添加）

```yaml
---
type: 事件记录
title: [根据内容拟定，如"董婷提成核算讨论"]
date: [从文件名或内容提取]
participants: [识别的参与者，如"老板（发言人1）、董婷（发言人3）"]
status: 二次转写中
tags:
  - [相关标签]
---

# [标题]

---
```

### 每段格式

```markdown
发言人 1  00:00:00
[原文内容保持不变]

**语义转写**：
[转写内容]

---
```

### 完整示例

```markdown
---
type: 事件记录
title: 董婷提成核算讨论
date: 2026-02-07
participants: 老板（发言人1）、董婷（发言人3）、同事（发言人2）
status: 二次转写中
tags:
  - 新会务公司
  - 提成核算
---

# 02-07 董婷提成核算讨论

---

发言人 1  00:00:00
这是，**，**办公室没球的走。门关上。你那个他们给你弄完了账没算清楚吗？

**语义转写**：
老板让无关人员离开、关门。然后问董婷：财务给你算的账，你觉得没算清楚？

---

发言人 3  00:00:12
咋回事？账没算清楚。

**语义转写**：
董婷：什么情况？账没算清楚。

---

发言人 1  00:00:18
但是他挨挨着给你看没嘛，关键是。

**语义转写**：
老板：关键是，他有没有一笔一笔给你看？

---

发言人 3  00:00:21
看了，但我没注意他那个，因为我因为我算的跟他那个算出来总金额差了有五。对折。

**语义转写**：
董婷：看了，但我没细看。因为我自己算的金额和他算的差了将近一半（对折）。

---

发言人 1  00:00:43
你要放**，滚啊滚远点哈，老子耍了。第一个我跟你说，听到我跟你讲，算对了的。听到我跟你说哈第一个是我们的开单金额减去我们的项目成本对吧啊就。等于毛利嘛，我一样要跟你讲清楚啊。好，然后完了我们的开单金额是要把那个扣出来的，差不多七个点的税费，这个你要明白。

**语义转写**：
老板（对同事）：不想参与就滚远点。

老板（对董婷）解释利润计算规则：
1. **毛利公式**：开单金额 - 项目成本 = 毛利
2. **税费扣除**：开单金额要先扣约 **7个点的税费**（比如开1000万的票，税已经交出去了）

---

发言人 3  00:02:42
二让你听我说啊按照你们的算法是毛利的百分之十其中的百分之六十来给我们项目组分。

**语义转写**：
董婷：让我说完。按照你们的算法：
- **项目奖金 = 毛利的 10%**
- **项目组分成 = 奖金的 60%**

---
```

### 格式要点

| 要点 | 说明 |
|------|------|
| **原文不动** | 保留发言人标记、时间戳、原始文字 |
| **转写紧跟** | `**语义转写**：` 紧跟原文后 |
| **分隔符** | 每段用 `---` 分隔 |
| **关键数字加粗** | 如 **7个点**、**10%**、**60%**、**3万** |
| **复杂内容用列表** | 规则、计算公式、多步骤用列表展示 |
| **不清晰标注** | 听不清的内容标注 `[内容不清晰]` |
| **多人对话** | 同一段转写中如涉及多人，分行标注 |

## 四、转写质量标准

### 必须保留

| 类型 | 示例 |
|------|------|
| **数字** | 50%、20%、3万块、5个人 |
| **人名** | 张兰心、钱程、刘睿、唐德龙 |
| **金额/比例** | 64:36、48%、12000元 |
| **关键结论** | "以后没有项目负责人了" |
| **承诺/决定** | "二六年必须分组" |

### 可以删除

| 类型 | 示例 |
|------|------|
| **语气词** | 嗯、啊、哦、是不是、对不对、明白没 |
| **重复表达** | "就是就是就是"、"对对对" |
| **无意义过渡** | "然后完了"、"好，继续来" |
| **口语填充** | "那个"、"这个"、"啥子" |

### 需要推断补全

| 场景 | 处理方式 |
|------|---------|
| **断句错误** | 合并成完整句子 |
| **指代不清** | 根据上下文补充主语 |
| **方言表达** | 转为普通话书面语 |

## 五、特殊处理

### 多人会议

- 识别发言人切换
- 保持发言人标记（发言人1、发言人2）
- 如有真名，在首次出现时标注

### 关键原话

如果原话很重要（表态、承诺、定性），用引用格式保留：

```markdown
**语义转写**：
老板明确：两个人没有办法协作，只能分组。

> **老板原话**："二六年那就是大碗一个组，先生一个组，只有这个样子。"
```

### 数字/比例讨论

如果是分钱、分工等涉及具体数字的讨论，必须精确：

```markdown
**语义转写**：
最终分配比例：客户经理团队48%，会场管理30%，设计22%。
- 48%：五件套（吃住行、接待、日程、大预算、签到）
- 30%：会场执行
- 22%：设计
```

## 六、用户校准指南

用户收到转写后，检查：

1. **数字对不对**？
2. **人名对不对**？
3. **意思有没有歪曲**？
4. **有没有漏掉关键信息**？

校准方式：
- 直接在原文里改
- 或告诉 AI："第3段转写有误，应该是..."

## 七、启动指令

### 用户启动方式

```
启动对文件 [文件路径] 的二次转写工作。

参与者识别：
- 发言人 1：[姓名/角色]
- 发言人 2：[姓名/角色]
- 发言人 3：[姓名/角色]
```

### AI 收到启动指令后

1. **读取文件**，确认文件存在、行数
2. **添加 YAML frontmatter**（如果没有）
3. **处理第一段**（约前 50-100 行），**直接用 StrReplace 工具写入原文件**
4. **提示用户校准**

⚠️ **关键**：必须使用 `StrReplace` 或 `Write` 工具直接修改用户指定的文件，**绝对不能创建新文件**！

### AI 第一轮回复示例

```
已启动二次转写。

文件：[输入] 02-07新公司董婷.md（共 1290 行）
参与者：老板（发言人1）、董婷（发言人3）、同事（发言人2）

已处理第一段（约 00:00:00 - 00:03:30），共 25 个发言段落，已写入原文件。

请在文件中校准：
1. 数字对不对？
2. 人名对不对？
3. 意思有没有歪曲？

校准完成后说"继续"，我处理下一段。
```

### 后续轮次

- 用户说"继续"→ AI 处理下一段，直接写入文件
- 用户说"第X段有问题，应该是..."→ AI 修正
- 全文处理完毕 → AI 将 `status: 二次转写中` 改为 `status: 已完成`

## 八、与数字真我的关系

二次转写是**材料预处理**，不是认知产出：

```
原始录音
    ↓
ASR 转写（Whisper/飞书妙记）
    ↓
【二次转写】← 本 SK
    ↓
用户校准
    ↓
数字真我复盘（提取画像/知识点/共识）
    ↓
入库
```

**二次转写不做复盘**，只做格式转换。复盘由数字真我负责。

| 模型 | 质量 | 成本 | 建议场景 |
|------|------|------|----------|
| **Gemini 3 Flash** | ★★★★☆ | 极低 | 日常转写首选，性价比最高 |

**推荐策略**：
- 常规转写 → Gemini 3 Flash（便宜、快、质量够用）
- 重要会议 → Gemini 3 Pro 或 Opus 4
- 在 Cursor 中操作 → Sonnet 4 已足够
